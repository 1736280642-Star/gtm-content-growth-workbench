# 工具契约、幂等、重试、补偿与并发控制

## 1. 设计结论

Agent 能否稳定运行，取决于工具是否具备正式契约，而不是模型是否“更聪明”。每个工具必须明确成功、失败、部分成功和未知结果，所有写操作必须可识别重复调用，所有重试必须建立在错误分类和幂等保障上。

## 2. 统一工具调用结构

### 2.1 ToolInvocationRequest

| 字段 | 含义 |
|---|---|
| invocation_id | 调用 ID |
| tool_id / tool_version | 工具契约 |
| workflow_run_id | 所属运行 |
| workflow_step_run_id | 所属步骤 |
| actor_type / actor_id | 调用主体 |
| input | 经过 schema 校验的输入 |
| input_hash | 输入校验值 |
| idempotency_key | 幂等键 |
| policy_decision_id | 权限决策 |
| approval_request_id | 可选人工确认 |
| deadline_at | 最晚完成时间 |
| trace_id | 跨服务追踪 ID |

### 2.2 ToolInvocationResult

| 字段 | 含义 |
|---|---|
| invocation_id | 对应调用 |
| status | succeeded、failed、partial、unknown |
| output | 结构化输出 |
| artifact_refs | 产物引用 |
| error_class / error_code | 错误分类 |
| retryable | 是否允许重试 |
| compensation_required | 是否需补偿 |
| started_at / ended_at | 时间 |
| provider_request_id | 脱敏后的外部请求引用 |

unknown 表示请求可能已到达外部系统但结果无法确认。unknown 不能直接重试写操作，必须先执行结果查询或人工核对。

## 3. 工具输出原则

- 结果必须是结构化对象，不以自然语言作为唯一成功依据。
- succeeded 必须同时返回预期产物或明确的无产物成功证明。
- partial 必须列出成功项、失败项和是否允许继续。
- failed 必须包含稳定 error_code 和用户可执行的 next_action。
- 不得在 error message 中包含密钥、完整 Prompt、原始请求体或敏感正文。
- 外部系统没有真实返回时，不得生成虚假的 request_id、URL 或数据。

## 4. 幂等设计

### 4.1 幂等键构成

建议逻辑：

    idempotency_key =
      tool_id
      + tool_version
      + business_object_id
      + normalized_input_hash
      + intended_effect

时间戳、随机数和 attempt_no 不进入业务幂等键，否则每次重试都会被视为新动作。

### 4.2 幂等处理

收到相同 idempotency_key 时：

- 已 succeeded：直接返回原结果引用。
- 正在 running：返回 in_progress，不启动第二次调用。
- 已 failed 且 retryable：创建新 attempt，沿用业务幂等键。
- 已 unknown：先调用 reconcile 工具确认外部结果。
- 输入不同但业务对象相同：拒绝覆盖，要求显式新版本或人工确认。

### 4.3 必须幂等的动作

- 创建 WorkflowRun。
- 生成 MonthlyStrategyPackageVersion 草稿。
- 生成 ContentMatrixVersion 和 ContentMatrixItem 草稿。
- 创建 BatchGenerationRun。
- 创建 GenerationRun。
- 生成或更新 PlatformDraftVariant 和 PublishSchedule。
- 创建 ExceptionItem。
- 创建 HumanApprovalRequest。
- 发送同类型提醒。
- 回填公开 URL。
- 激活治理版本。
- 触发外部 Provider 生成。

## 5. 错误分类

| error_class | 示例 | 默认处理 |
|---|---|---|
| pending_config | Provider、Embedding、渠道源未配置 | waiting_config，不重试 |
| validation_error | schema、参数、版本不合法 | 阻断并返回修改指引 |
| policy_denied | 权限或对象范围不允许 | 拒绝并审计 |
| business_blocked | 证据不足、硬规则阻断 | 异常分流或人工处理 |
| conflict | 锁冲突、版本冲突、人工修改冲突 | 重新读取状态 |
| rate_limited | 外部限流 | 按 Retry-After 重试 |
| transient_external | 网络、临时 5xx | 指数退避重试 |
| timeout_unknown | 请求超时且结果未知 | 先 reconcile |
| permanent_external | 4xx、资源不存在 | 不自动重试 |
| internal_bug | 未分类异常、契约破坏 | 停止并告警 |

不得根据错误文本临时猜测是否重试。错误分类由工具适配层产生。

## 6. 重试策略

### 6.1 允许自动重试

- 无副作用的只读调用。
- 有幂等键且外部系统支持幂等的写调用。
- 可通过本地唯一约束保证不重复的内部写调用。
- 明确返回 rate_limited 或 transient_external 的调用。

### 6.2 禁止自动重试

- policy_denied、validation_error、business_blocked、pending_config。
- 已发生人工编辑冲突。
- 正式发布等 L4 动作。
- timeout_unknown 且尚未 reconcile。
- 外部系统不支持幂等且无法查询结果的写操作。

### 6.3 建议参数

- 初始间隔：2 秒。
- 指数退避：2、5、15、45 秒，并加入抖动。
- 普通内部调用最多 3 次。
- 外部 Provider 最多 2 次，避免重复成本。
- 超过最大次数后创建 ExceptionItem。
- Retry-After 优先于本地退避参数。

具体参数进入 RetryPolicy 配置，不硬编码在 Agent Prompt。

## 7. 超时、熔断与降级

每个 ToolDefinition 必须有连接超时、执行超时和总 deadline。

当同一 Provider 在滑动窗口内连续失败达到阈值：

1. 熔断器打开。
2. 新请求直接返回 provider_unavailable。
3. 工作流进入 waiting_external 或使用已批准的备用 Provider。
4. 备用 Provider 必须在运行启动前属于允许配置，不能由 Agent 临时选择。
5. 熔断恢复需通过健康探测，不由聊天指令强行关闭。

降级不能绕过质量标准。若备用模型不满足版本准入，宁可显示等待，也不能伪装完成。

## 8. 补偿机制

补偿适用于已经产生可逆副作用的动作：

| 原动作 | 补偿动作 |
|---|---|
| 创建重复内部草稿 | 标记重复并撤销有效引用 |
| 创建错误确认请求 | 撤销请求并创建新版本 |
| 错误绑定 ArtifactReference | 关闭旧引用并写入新引用 |
| 发送错误内部提醒 | 发送更正提醒 |
| 激活错误治理版本 | 通过人工批准回滚到上一可用版本 |

补偿不是删除历史。原动作、补偿动作、原因和执行人均保留。

正式外部发布不属于可由 Agent 补偿的动作，必须由人处理撤回、更正或重发。

## 9. 并发控制

### 9.1 WorkflowLock 字段

- lock_key
- resource_type / resource_id
- owner_worker_id
- workflow_run_id
- acquired_at
- expires_at
- heartbeat_at
- status

### 9.2 控制规则

- Worker 领取 ready 步骤时使用原子 compare-and-set。
- 同一 workflow_step_run_id 同时只允许一个 active attempt。
- 草稿进入人工编辑后，自动写入工具必须获取 edit_lock，默认会失败并转冲突队列。
- 治理版本激活使用串行锁和数据库唯一约束。
- 定时任务使用 schedule_fire_key 防止服务重启后重复触发。

## 10. 成本与调用预算

对 Provider、Embedding、重排和评测模型设置：

- 单运行调用次数上限。
- 单任务 Token 或费用预算。
- 单产品和单日速率上限。
- 超预算后的 waiting_human 或 blocked 策略。

预算状态必须进入运行详情。Agent 不允许为追求“完成”无限重试或自动切换到未批准的高成本模型。

## 11. 安全与数据边界

- sensitive_fields 在进入日志前统一脱敏或丢弃。
- 工具输入只传完成任务所需的最小字段。
- 知识库检索结果继续执行对象权限过滤。
- Prompt 和系统策略以版本引用传递，普通调用记录不保存完整正文。
- 外部 Provider 请求必须经过允许字段白名单。
- 任何工具不得返回 .env、Token、私有链接或客户原始数据。

## 12. 对使用者的影响

用户会看到更明确的错误类型：

- “缺少配置，配置后可继续。”
- “网络临时失败，系统将在 15 秒后重试。”
- “请求结果未知，正在核对，暂不重复执行。”
- “正文已被人工修改，系统不会覆盖，请选择保留或创建新版本。”

这比统一显示“生成失败，请重试”更可信，也能显著减少重复成本和数据污染。

## 13. 验收标准

- 相同幂等键重复 20 次只产生一个有效副作用。
- Worker 在调用中崩溃后可通过 ToolInvocationRecord 判断是否重试。
- timeout_unknown 不会直接触发第二次外部写操作。
- pending_config 不进入无意义重试循环。
- 人工编辑锁能阻止自动覆盖。
- 并发激活治理版本只有一个成功。
- 熔断时不会临时切换到未批准 Provider。
- 所有失败都有 error_class、error_code 和 next_action。
- 调用日志中不存在密钥、完整 Prompt 和原始模型 trace。
