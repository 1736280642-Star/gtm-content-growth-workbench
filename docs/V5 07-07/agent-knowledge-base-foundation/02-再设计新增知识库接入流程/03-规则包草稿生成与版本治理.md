# 规则包草稿生成与版本治理

## 1. 目标

把已完成来源绑定、事实抽取和初步复核的 `ProductClaim`，转换为可审阅的 `ProductExpressionRulePackage` 新版本草稿。

核心原则：规则包不是资料摘要，也不是 Prompt；它是决定内容生产“允许怎么说、必须带什么条件、什么必须拦截”的治理真源。

## 2. 输入与输出

### 2.1 输入

- 产品实体和当前生效规则包。
- 新增资料及其证据等级。
- 新增、加强、降级、确认、冲突和替代事实。
- 已识别的限制、条件和资料缺口。
- 第一阶段通用硬拦截规则。

### 2.2 输出

- 新规则包草稿。
- 与当前生效版本的字段级差异。
- 每项变更关联的 `claimId / sourceId`。
- 待确认角色和原因。
- 受影响的蒸馏词、内容任务和已发布内容范围。

## 4. 事实到规则的转换

### 4.1 产品身份

只有 `supported/conditional` 且满足最低证据的身份事实可以写入。名称、品牌归属、官方主体、官网和产品类别发生变化时，默认要求业务确认；官方主体变化还可能要求法务确认。

### 4.2 能力状态

每项能力单独保存：

```yaml
capabilityId: "cap-..."
name: ""
status: "confirmed"
scope: ""
conditions: []
limitations: []
applicableVersion: ""
evidenceClaimIds: []
```

允许状态：

- `confirmed`
- `conditional`
- `beta`
- `planned`
- `deprecated`
- `disputed`

`beta / planned / disputed` 不能进入无条件允许表达。

### 4.3 允许表达

允许表达必须说明：

- 推荐写法。
- 适用场景和渠道。
- 对应事实和证据。
- 是否要求引用官网。
- 是否允许语义改写。

### 4.4 条件表达

条件表达必须包含：

- 允许写法。
- 成立条件。
- 必须披露的限制。
- 最低证据等级。
- 适用版本或部署范围。
- 不满足条件时的降级写法。

### 4.5 禁止表达与硬拦截

自然语言规则必须转换为可执行结构：

```yaml
ruleId: "block-..."
riskType: "performance"
description: "无测试条件不得使用绝对延迟承诺"
matchMode: "hybrid"
patterns: []
semanticIntent: ""
structuredCondition:
  claimType: "performance_metric"
  missingFields: ["testEnvironment", "percentile", "productVersion"]
action: "block"
fallbackAction: "remove_metric_and_rewrite_qualitatively"
allowedException: ""
approvalRoles: ["technical_owner"]
evidenceClaimIds: []
```

`keyword` 只适合明确禁词；产品混淆、事实扩大和条件丢失需要语义或结构化校验。规则包必须允许多种匹配方式并存。

## 5. 版本差异类型

每一项 `changeSet` 使用以下类型：

- `added`
- `strengthened`
- `weakened`
- `condition_added`
- `condition_removed`
- `status_changed`
- `evidence_upgraded`
- `evidence_downgraded`
- `conflict_detected`
- `deprecated`
- `removed`
- `no_material_change`

差异记录：

```yaml
changeId: "chg-..."
section: "capabilities"
fieldPath: "capabilities.cap-001.status"
changeType: "weakened"
before: "confirmed"
after: "conditional"
reason: "新部署文档增加了外部 Provider 条件"
claimIds: []
sourceIds: []
riskLevel: "high"
requiredRoles: ["technical_owner", "privacy_owner"]
```

页面必须优先展示“新增了什么、哪些表达被加强、哪些被降级、哪些冲突尚未解决”，不能只展示完整新旧 JSON。

## 6. 版本号规则

建议使用语义化版本：

- `major`：产品身份、核心类别、关键安全/隐私边界发生重大改变。
- `minor`：新增正式能力、渠道规则、证据要求或较大规则组。
- `patch`：措辞澄清、证据补充、非实质性规则修正。

草稿形式：`1.2.0-draft.1`。正式生效后为 `1.2.0`。

版本号由系统建议，确认人可以调整，但必须记录原因。

## 7. 草稿生成策略

### 7.1 已有产品新增资料

以当前 `active` 版本为基线，只生成差异草稿。未受影响字段保持不变，避免模型重写整个规则包造成无意义变化。

### 7.2 新产品

套用第一阶段通用模板生成完整草稿，但只填入有证据的内容；缺失字段进入 `evidenceGaps`，不得用模型常识补全。

### 7.3 关联已有规则包

当新知识库选择“关联已有规则包”时：

- 新资料仍需完成来源分类和事实抽取。
- 事实进入该产品的候选事实池。
- 只有发现实质变化才生成新版本草稿。
- 没有实质变化时记录 `no_material_change`，不制造空版本。

### 7.4 多知识库共同支撑

规则包属于产品，不属于单个知识库。多个知识库可以共同提供证据，规则包保存全部 `linkedKnowledgeBaseIds` 和逐条事实来源，不能继续把规则包永久挂在某一个知识库对象下面。

## 8. 状态与审批

规则包草稿可以同时需要多个角色确认。`pendingRoles` 是集合，不应通过连续覆盖状态丢失审批信息。

推荐模型：

```yaml
status: "draft_pending_confirmation"
pendingRoles: ["business_owner", "technical_owner", "privacy_owner"]
approvals:
  - role: "business_owner"
    status: "approved"
  - role: "technical_owner"
    status: "pending"
  - role: "privacy_owner"
    status: "changes_requested"
```

为兼容第一阶段状态名称，页面可以显示 `draft_pending_privacy_confirmation` 等主状态，但数据层应保留多角色审批明细。

## 9. 生效条件

规则包只有满足以下条件才可变为 `active`：

1. 产品身份必填项完整。
2. 核心能力有状态、范围和证据。
3. 强事实满足最低证据等级。
4. 阻断级冲突已裁决或采用明确保守口径。
5. 硬拦截规则具有可消费结构。
6. 所有必需角色完成确认。
7. 差异和影响范围已生成。
8. 关联来源没有处于隔离或失效状态。
9. 月度矩阵适用范围已经生成并通过业务负责人确认。

## 10. 回滚与影响分析

回滚不是删除新版本，而是：

1. 将目标旧版本重新设为 `active`。
2. 当前版本标记 `rolled_back`。
3. 记录原因、操作者、时间和关联事件。
4. 标记使用新版本生成的任务、草稿和发布内容。
5. 检查新版本创建或启用的蒸馏词是否需要停用。
6. 通知第三阶段刷新可检索规则版本和 EvidencePack 缓存。

## 11. 现有代码迁移建议

现有 `ProductExpressionRuleDraft` 的：

- `doExpressions` 可迁移到 `allowedExpressions`。
- `dontExpressions` 可迁移到 `blockedExpressions`，但必须补匹配方式和动作。
- `boundaryNotes` 可迁移到条件表达或限制说明。
- `distilledTermSuggestions` 保留为候选，不直接视为已生效蒸馏词。
- `previousSnapshot` 保留作为兼容快照，同时新增正式版本实体。

不要一次删除旧字段；先做双写和读取兼容，待 V5 数据迁移完成后再移除。

## 12. 验收标准

1. 新资料不会直接修改当前 `active` 规则包。
2. 草稿只修改受影响字段，不无意义重写整包。
3. 每项变更可追溯到事实和来源。
4. 降级、冲突和删除比新增能力更醒目。
5. 多角色审批不会被压缩成一个模糊 `draft` 状态。
6. 回滚后任务、蒸馏词和第三阶段检索影响可识别。
7. active 规则包能够直接输出月度生产准入范围，草稿规则包不能进入月度内容生产池。
