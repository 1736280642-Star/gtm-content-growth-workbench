# 资料缺口、来源冲突与人工确认

## 1. 目标

把“资料不足”拆成可处理的业务状态，并明确哪些问题由系统降级、哪些进入补资料、哪些必须由特定责任人裁决。

资料缺口、来源冲突和解析失败不能统一显示成一个红色警告，因为三者的原因、风险和处理动作不同。

## 2. 三类问题边界

### 2.1 资料缺口 `EvidenceGap`

系统没有足够资料支撑某个必需字段或表达。

### 2.2 来源冲突 `ClaimConflict`

系统拥有多个来源，但它们对同一事实的值、状态、范围、版本或时间给出不一致结论。

### 2.3 流程异常 `IngestionIssue`

抓取失败、解析失败、配置缺失、敏感资料、实体歧义等导致流程无法继续。这不代表产品事实本身缺失。

## 5. 严重度与系统动作

| 严重度 | 判断 | 系统动作 |
| --- | --- | --- |
| `info` | 不影响当前主要表达 | 记录并允许继续 |
| `warning` | 可使用保守或降级表达 | 草稿带条件，提示补资料 |
| `high` | 影响重要能力或渠道使用 | 进入人工确认，不自动生效 |
| `blocking` | 产品身份、强事实、安全隐私或来源合法性无法确认 | 阻断规则包或对应表达 |

阻断应尽量精确到字段或表达。缺少性能报告时可以阻断性能数字，但不应阻断已有充分证据的产品定义。

## 6. 冲突分类

- `value_conflict`：数字、名称、数量不同。
- `status_conflict`：当前、Beta、规划、下线不一致。
- `scope_conflict`：所有部署与特定部署不一致。
- `version_conflict`：不同版本能力不同。
- `time_conflict`：旧资料与新资料不一致但替代关系不清。
- `ownership_conflict`：产品、公司或第三方能力归属不一致。
- `privacy_conflict`：官网承诺与隐私/数据流资料不一致。
- `case_conflict`：案例数字、匿名状态或授权范围不一致。
- `comparison_conflict`：竞品结论的时间和版本不一致。

## 7. 冲突组结构

```yaml
conflictGroupId: "conflict-..."
productId: ""
conflictType: "privacy_conflict"
subject: "第三方 AI 处理器是否参与"
claimIds: []
sourceIds: []
preferredTemporaryClaimId: ""
temporaryPolicy: "use_more_conservative_claim"
severity: "blocking"
requiredRoles: ["technical_owner", "privacy_owner"]
status: "open"
resolution:
  selectedClaimId: ""
  applicableVersion: ""
  reason: ""
  decidedBy: []
  decidedAt: ""
```

系统可以建议“权威更高、时间更近、范围更明确”的来源，但不能自动裁决安全、隐私、合规、客户效果和重大产品能力冲突。

## 8. 临时口径

冲突未解决时只能采用以下策略：

- `use_more_conservative_claim`
- `downgrade_to_conditional`
- `remove_metric`
- `limit_to_specific_version`
- `limit_to_specific_deployment`
- `mark_as_planned_or_unverified`
- `block_public_expression`

不得采用“选择营销效果更好的一条”。

## 9. 人工确认角色

| 角色 | 主要确认内容 |
| --- | --- |
| `business_owner` | 产品身份、目标用户、核心场景、主价值 |
| `product_owner` | 当前能力、版本状态、产品边界 |
| `technical_owner` | 架构、集成、兼容、性能、部署条件 |
| `security_owner` | 安全控制、检出率、攻击覆盖、风险承诺 |
| `privacy_owner` | 数据流、第三方处理器、数据出域和隐私口径 |
| `legal_owner` | 公司主体、资质、合规、合同和竞品风险 |
| `delivery_owner` | 客户案例、实施范围、POC 与交付事实 |
| `knowledge_manager` | 来源分类、证据关联、资料质量和流程完整性 |

一个人可以在组织中承担多个角色，但系统仍需记录其以哪个角色完成确认。

## 10. 确认单元

支持三种确认粒度：

1. `claim`：确认一条事实。
2. `change`：确认规则包的一项变更。
3. `package`：在所有必要项完成后确认整个版本生效。

不推荐只提供“整包确认”按钮。高风险项必须逐项展示，整包确认只负责最后生效，不代表跳过明细审查。

## 11. 审批动作

- `approve`
- `approve_with_conditions`
- `request_changes`
- `reject`
- `request_more_evidence`
- `accept_conservative_wording`
- `defer`

每次动作必须填写或自动记录：

- 操作对象。
- 操作前后状态。
- 操作角色。
- 原因。
- 证据或补充资料。
- 时间。
- 是否影响规则包、蒸馏词和第三阶段检索。

## 12. 异常队列

统一异常队列按动作组织，而不是按技术错误堆日志：

- 待确认产品实体。
- 待分类资料。
- 待补正式证据。
- 待裁决来源冲突。
- 待确认高风险规则变更。
- 待处理敏感资料。
- 待重试解析或抓取。
- 待确认蒸馏词和用户问题。

队列默认展示：业务影响、阻断范围、责任角色、建议动作、来源和进入时间。原始模型日志留在开发或 AI 治理视图。

## 13. 三产品首批缺口模板

### Pharaoh Command

- 正式主体与产品归属。
- 兼容矩阵。
- 数据流和 Provider 边界。
- Workflow 写权限、审批和回滚。
- 性能和客户效果测试口径。

### Noteflow

- 官网与隐私政策的数据处理冲突。
- 公有版和私有版差异。
- 第三方 AI/视觉/ASR 处理器。
- 规划功能与当前功能清单。
- 复杂文档解析和引用精度评测。

### 唯客 AI 护栏

- PII 支持数量冲突。
- 延迟测试条件。
- Dify 应用类型和版本兼容范围。
- 云扫描与数据不出域边界。
- 检出率、误报率、客户数量和资质证据。

## 14. 验收标准

1. 缺口、冲突和流程异常分别呈现。
2. 缺口可以精确阻断某类表达，而不是默认阻断整个产品。
3. 每个高风险问题有责任角色和下一步动作。
4. 冲突未解决时自动采用保守策略或阻断。
5. 人工确认保留角色、原因、时间和证据。
6. 普通内容发布人员不会看到或操作治理确认入口。
