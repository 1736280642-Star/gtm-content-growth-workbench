# 阶段二：新增知识库接入最佳设计总方案

## 1. 文档定位

本阶段负责把第一阶段已经定义的 `ProductExpressionRulePackage`，转化为一条可执行、可审计、可人工确认的新增资料接入链路。

它解决的不是“如何上传一个文件”，而是：当官网、产品文档、案例、FAQ、技术资料或历史文章进入系统后，系统如何判断资料属于谁、能证明什么、与现有事实是否冲突、应该生成怎样的规则包变更草稿，以及哪些问题必须交给人确认。

阶段二的最终结果不是“知识库里多了一批文本”，而是形成五类可被后续阶段稳定消费的治理资产：

1. 可追溯的来源资料 `SourceAsset`。
2. 绑定原文证据的原子事实 `ProductClaim`。
3. 不覆盖生效版本的规则包草稿 `RulePackageDraft`。
4. 明确的资料缺口与来源冲突 `EvidenceGap / ClaimConflict`。
5. 可追溯的蒸馏词与用户问题候选 `TermCandidate / QuestionCandidate`。

## 2. 核心判断

最佳路径不是：

```text
上传资料 -> AI 总结 -> 生成允许表达/禁止表达 -> 人确认
```

因为这条路径把“资料原文、产品事实、表达规则”混成了一层，后续无法回答某条规则来自哪里、为什么发生变化、冲突时依据什么裁决。

本阶段采用四层模型：

```text
来源层 SourceAsset
-> 事实层 ProductClaim
-> 治理层 ProductExpressionRulePackage
-> 候选层 TermCandidate / QuestionCandidate
```

四层必须分开：

- 来源层保存“资料是什么、从哪里来、当前是否有效”。
- 事实层保存“资料明确支持了什么主张，以及原文定位”。
- 治理层决定“这些事实允许如何表达、何时必须降级或拦截”。
- 候选层把事实转成后续内容规划可使用的蒸馏词和用户问题，但不能反向修改产品事实。

## 3. 阶段边界

### 3.1 本阶段负责

1. URL、文档、手工文本和批量抓取结果的统一接入。
2. 导入方式、业务资料类型、证据等级、生命周期和公开边界识别。
3. 产品实体识别、已有产品匹配、新产品候选和跨产品资料分流。
4. 原子事实抽取、原文证据定位、置信度和适用版本记录。
5. 与当前生效规则包及既有事实的差异、冲突和缺口分析。
6. 生成新的规则包草稿，不覆盖当前生效版本。
7. 生成可追溯的蒸馏词和用户问题候选。
8. 按风险和责任人进入业务、技术、安全、隐私、法务或交付确认。
9. 为第三阶段输出通过准入的来源、事实、权威等级和治理状态。

### 3.2 本阶段不负责

1. 不定义向量切片、Embedding、混合召回和 Top-K 排序；这些属于第三阶段。
2. 不生成正式正文，也不修改渠道 Prompt；这些属于第四阶段。
3. 不允许 AI 自动批准产品事实、规则包、合规承诺和强效果数字。
4. 不允许未确认资料直接进入生产检索。
5. 不自动创建正式产品实体；只能创建待确认实体候选。
6. 不因新资料删除旧证据；旧资料通过失效、替代或归档状态保留审计链。

## 4. 第一阶段输入与阶段二补充

第一阶段已经提供：

- 产品身份与不可混淆实体。
- 能力状态与条件表达。
- 允许表达、禁止表达和渠道边界。
- 证据等级与最低证据要求。
- 资料缺口、冲突处理、版本和回滚原则。
- Pharaoh Command、Noteflow、唯客 AI 护栏三个真实审计样本。

阶段二需要把这些人类可读规则转成结构化数据契约，并补充：

1. 每条事实的来源、原文片段和定位。
2. 资料类型与证据角色的独立分类。
3. 产品实体识别和歧义处理。
4. 新资料与已有事实的合并、冲突和替代规则。
5. 不同缺口的严重度、阻断级别和责任人。
6. 规则包草稿与当前生效版的字段级差异。
7. 蒸馏词和用户问题候选到来源事实的追溯关系。

## 5. 最佳业务流程

```text
创建导入批次
-> 安全与公开边界检查
-> 内容解析与正文完整性检查
-> 资料类型、权威等级和生命周期识别
-> 产品实体匹配与人工校正
-> 原子事实抽取与证据定位
-> 事实去重、冲突和时效判断
-> 资料缺口检查
-> 生成规则包变更草稿
-> 生成蒸馏词和用户问题候选
-> 责任人分级确认
-> 规则包生效 / 退回补资料 / 保持阻断
-> 计算月度生产准备度与月度矩阵适用范围
-> 产品进入月度内容生产池或月度准入异常队列
-> 输出第三阶段可用来源与事实
```

### 5.1 七个强制闸门

| 闸门 | 核心问题 | 不通过处理 |
| --- | --- | --- |
| G0 安全准入 | 是否包含密钥、隐私、后台数据或未经授权客户资料 | 隔离，禁止继续处理 |
| G1 解析质量 | 是否获得完整、可定位、非错误页的正文 | 重试、人工补充或标记解析失败 |
| G2 类型与实体 | 资料是什么、属于哪个产品，是否存在歧义 | 人工校正，不能自动归属 |
| G3 事实与证据 | 每条事实是否有原文、来源和适用边界 | 降级为候选或标记证据不足 |
| G4 冲突与缺口 | 是否与现有事实冲突，是否缺强事实证据 | 保守口径、补资料或分派确认 |
| G5 规则包确认 | 谁确认了哪些变更，是否满足生效条件 | 保持草稿，不进入生产使用 |
| G6 月度生产准入 | 规则包和证据能支撑哪些内容类型、渠道和月度配额 | 限制矩阵范围、进入缺口队列或阻止入池 |

## 6. 状态模型

### 6.1 导入批次 `IngestionBatch`

```text
draft
-> parsing
-> classifying
-> awaiting_entity_review
-> extracting_claims
-> reviewing_conflicts
-> generating_rule_draft
-> awaiting_approval
-> completed
```

异常状态：

- `blocked_sensitive_data`
- `blocked_entity_ambiguous`
- `blocked_evidence_conflict`
- `partial_failed`
- `failed`
- `cancelled`

批次状态只表示流程位置，不能代替单份资料、单条事实和规则包自身的状态。

### 6.2 来源资料 `SourceAsset`

- `pending_parse`
- `parsed`
- `review_required`
- `approved_for_claim_extraction`
- `isolated`
- `superseded`
- `deprecated`
- `parse_failed`

### 6.3 原子事实 `ProductClaim`

- `candidate`
- `supported`
- `conditional`
- `disputed`
- `rejected`
- `superseded`
- `expired`

### 6.4 规则包 `ProductExpressionRulePackage`

正式状态统一为：

- `draft_pending_business_confirmation`
- `draft_pending_technical_confirmation`
- `draft_pending_security_confirmation`
- `draft_pending_privacy_confirmation`
- `draft_pending_legal_confirmation`
- `draft_pending_delivery_confirmation`
- `active`
- `deprecated`
- `rolled_back`
- `archived`

现有代码中的 `draft / active / archived` 可以作为兼容状态，但后续实现必须增加 `pendingRole` 或迁移到以上正式状态，不能继续用一个 `draft` 掩盖所有审批责任。

## 7. 自动化与人工判断边界

### 7.1 系统可以自动完成

- 解析、去模板、Hash 去重和错误页识别。
- 资料类型、产品实体和事实候选建议。
- 原文引用定位、证据等级建议和置信度。
- 已有事实匹配、字段级差异、冲突候选和缺口候选。
- 规则包草稿、蒸馏词候选和用户问题候选。
- 根据风险类型分派确认角色。

### 7.2 必须由人决定

- 新产品实体是否建立。
- 模糊资料最终属于哪个产品。
- 冲突事实采用哪个口径及适用版本。
- 性能、客户效果、数据流、安全、隐私和合规主张是否可对外使用。
- 规则包草稿是否生效。
- 被系统标记为阻断的资料是否允许例外使用。

## 8. 关键设计原则

1. **先证据，后事实，再规则**：不能直接从整篇文章生成规则包。
2. **逐条追溯**：每条强表述必须关联 `claimId` 和至少一个 `sourceLocator`。
3. **默认保守**：来源冲突、版本未知和证据不足时，不采用更有营销力的口径。
4. **新资料不覆盖旧版本**：只生成新草稿和差异，人工确认后生效。
5. **资料用途不等于资料事实**：用户填写的用途只影响流程和路由，不能作为证据。
6. **导入方式不等于资料类型**：URL、文件是技术入口；官网、案例、测试报告是业务证据类型。
7. **候选不等于生效**：蒸馏词、用户问题、能力和规则建议都必须保留候选状态。
8. **无真实配置不伪造结果**：解析、模型、抓取或外部服务缺失时返回 `pending_config` 或明确失败状态。

## 10. 阶段完成标准

第二阶段只有同时满足以下条件才算完成：

1. 新增资料能区分导入方式、资料类型、证据等级、生命周期和公开边界。
2. 产品实体识别结果可人工校正，歧义不会静默归属。
3. 每条产品事实都能回查到资料、原文定位和适用版本。
4. 新资料与已有事实的新增、加强、降级、冲突、替代和无变化可区分。
5. 规则包只生成草稿，不覆盖当前生效版本。
6. 缺口和冲突有严重度、责任人和下一步动作。
7. 蒸馏词和用户问题候选能追溯到 `claimId` 和 `sourceId`。
8. 人工确认记录包含确认人、角色、原因、时间和确认范围。
9. 未确认或被隔离的资料不会进入第三阶段生产检索。
10. Pharaoh Command、Noteflow、唯客 AI 护栏三个样本均通过端到端验收。
11. 每个 active 规则包都有月度矩阵适用范围和 MonthlyProductionReadiness。
12. 不满足月度生产准入的产品不会进入 MonthlyPlan 候选池。
