# 通用产品表达规则模板

## 1. 模板用途

本模板用于任何新产品或新知识库进入工作台时生成 `ProductExpressionRulePackage` 草稿。

模板解决的是“产品能怎么说、不能怎么说、必须凭什么证据说”。它不负责切片和召回，也不能替代产品文档、合同、技术规格、安全评测或法律意见。

固定治理原则：

1. 知识库负责提供事实候选，规则包负责决定表达边界。
2. 新资料只能生成规则包草稿，不能自动生效。
3. 官网营销页不自动等于强事实证据。
4. 历史文章只能作为表达和问题背景参考，不能单独证明产品能力。
5. 页面之间存在冲突时，采用更保守口径并进入人工确认。
6. 缺证据时降级、拦截或补资料，不能让模型自行补全。

## 3. 产品身份与定位

| 字段 | 填写内容 | 必填 |
| --- | --- | --- |
| 产品名称 | 统一对外名称和大小写 | 是 |
| 品牌归属 | 品牌、公司或产品线 | 是 |
| 官方主体 | 对外签约或运营主体 | 是 |
| 官网 URL | 产品主官网，不填搜索页或聚合页 | 是 |
| 产品类别 | 用户能够理解的类别 | 是 |
| 一句话定义 | 主体 + 服务对象 + 核心机制 + 解决问题 | 是 |
| 目标用户 | 决策者、使用者、维护者分别填写 | 是 |
| 核心场景 | 3-8 个可验证场景 | 是 |
| 核心问题 | 用户原有工作流中的真实阻塞 | 是 |
| 主卖点 | 只写有证据的差异点 | 是 |
| 竞品边界 | 可比较维度和不可做的排他判断 | 是 |
| 相似实体 | 容易混淆的品牌、产品和通用概念 | 是 |
| 当前版本 | 能力对应的软件或服务版本 | 建议 |

一句话定义模板：

```text
{产品名称} 是面向 {目标用户} 的 {产品类别}，通过 {核心机制}，帮助其解决 {核心问题}。
```

## 4. 能力状态表

每项能力必须单独记录，不允许用一段营销文案覆盖全部能力。

| capabilityId | 能力名称 | 状态 | 能力边界 | 适用版本 | 证据 ID | 备注 |
| --- | --- | --- | --- | --- | --- | --- |
|  |  | `confirmed / conditional / beta / planned / deprecated / disputed` |  |  |  |  |

状态解释：

- `confirmed`：当前正式资料和实际版本均可证明。
- `conditional`：只在特定部署、版本、配置或权限下成立。
- `beta`：可以说明测试状态，不得作为稳定承诺。
- `planned`：只能写“规划中”。
- `deprecated`：不得继续用于新内容。
- `disputed`：来源冲突，等待人工裁决。

## 7. 禁止表达与硬拦截

### 7.1 通用硬拦截

以下规则适用于所有产品：

1. 无证据的“第一、唯一、最佳、领先、全面超越”。
2. 无测试条件的“100%、零遗漏、零延迟、永不出错”。
3. 将“支持、帮助、降低风险”扩大为“保证结果”。
4. 将规划中、Beta、Demo 或 POC 能力写成正式生产能力。
5. 将匿名、模拟或代表性场景写成真实客户案例。
6. 将外部产品、开源组件或合作伙伴能力归为本产品原生能力。
7. 无数据流证据的“数据绝不出网、完全本地、零泄漏”。
8. 无法律或资质证据的“保证合规、通过某级认证”。
9. 无版本和公开来源的竞品贬损或排他对比。
10. 将历史博客、SEO 文章或模型生成内容当作产品事实真源。

## 8. 证据等级与引用规则

### 8.1 证据等级

| 等级 | 来源 | 可支撑内容 |
| --- | --- | --- |
| A | 合同附件、正式技术规格、部署架构、评测报告、有效资质 | 强事实、性能、数据流、兼容性、合规资质 |
| B | 当前官网产品页、正式功能文档、版本发布记录 | 产品定义、正式能力和场景 |
| C | 经授权客户案例、POC 报告、FAQ、售前说明 | 场景、条件化效果和常见问题 |
| D | 历史博客、渠道文章、销售材料 | 问题背景、表达结构和候选关键词 |
| E | 模型生成内容、未确认草稿、转载或无来源材料 | 不得支撑产品事实 |

### 8.2 表达与最低证据

| 表达类型 | 最低等级 | 额外要求 | 无证据处理 |
| --- | --- | --- | --- |
| 产品定义 | B | 当前有效页面 | 降级为通用类别描述 |
| 功能能力 | B | 标注版本或状态 | 标记需确认 |
| 性能数字 | A | 环境、样本、方法、分位值、日期 | 删除数字 |
| 客户效果 | C | 客户授权、基线、时间和测量口径 | 不写客户事实 |
| 数据流和隐私 | A | 数据流图、外部处理器和部署模式 | 使用保守口径 |
| 安全与合规 | A | 有效证书或正式评估 | 写“帮助实施控制” |
| 竞品比较 | A / B | 同期版本、同一维度、公开来源 | 删除绝对结论 |

## 9. 来源冲突处理

当两个来源对功能状态、数据路径、数字或版本描述不一致时：

1. 创建 `conflictGroupId`，保留全部来源。
2. 默认采用更保守的表达。
3. 禁止 Agent 自行选择更有营销力的版本。
4. 分派给产品、技术、安全、法务或交付负责人确认。
5. 确认后记录裁决理由和适用版本，不删除旧证据。

## 10. 渠道表达边界

| 渠道 | 主要任务 | 允许强度 | 必须保留 | 禁止倾向 |
| --- | --- | --- | --- | --- |
| 官网博客 | 建立正式产品认知 | 中高，但需证据 | 产品主体、机制、来源 | 空泛排名、无证据数字 |
| CSDN / 掘金 | 解释工程机制与落地 | 中 | 架构、条件、限制、验证 | 纯营销稿、伪代码式能力 |
| 公众号 / 知乎 / 头条 | 建立业务判断 | 中 | 真实问题、机制、适用边界 | 功能堆叠和恐惧营销 |
| GEO 信源站 | 建立机器可识别实体 | 克制 | 实体定义、能力、场景、官网 | 口号、重复关键词、模糊归因 |

渠道只能改变表达形式，不能改变产品事实、能力状态和证据要求。

### 10.1 月度矩阵适用范围

规则包必须明确本产品在月度内容矩阵中的可生产范围：

- 哪些内容类型可以直接进入矩阵。
- 哪些内容类型必须先补案例、数字、官网或技术证据。
- 哪些内容类型因产品阶段或合规风险禁止进入矩阵。
- 哪些渠道允许使用，哪些渠道需要更严格的表达限制。
- 当前证据最多支持多少月度配额，不能用配额目标倒逼虚构角度。

月度矩阵只能引用 active 版本。规则包回滚、证据失效或适用范围收紧时，必须重新计算受影响矩阵项的 Evidence Gate。

## 11. 资料缺口

每个产品至少检查：

- [ ] 正式产品主体是否确认。
- [ ] 当前版本和已上线能力是否确认。
- [ ] 官网、产品文档和实际版本是否一致。
- [ ] 规划中、Beta 和正式能力是否区分。
- [ ] 部署模式、数据流和第三方处理器是否清楚。
- [ ] 性能数字是否有测试报告。
- [ ] 客户案例是否获得公开授权。
- [ ] 竞品比较是否有同期来源。
- [ ] 安全、隐私和合规表述是否经责任人确认。
- [ ] 不可混淆产品实体是否建立。

输出状态：

- `ready_for_content`
- `ready_with_conditions`
- `needs_more_evidence`
- `needs_business_confirmation`
- `needs_technical_confirmation`
- `needs_legal_confirmation`
- `blocked`

## 12. 生效、变更与回滚

### 12.1 生效条件

1. 产品身份字段完整。
2. 核心能力逐项标记状态。
3. 硬拦截规则可被规则引擎消费。
4. 强事实关联到明确证据。
5. 来源冲突已解决或使用保守口径。
6. 产品负责人完成确认；涉及技术、安全、隐私和法律承诺时增加对应负责人确认。

### 12.2 变更要求

- 新知识库只生成新草稿，不覆盖当前生效版。
- 变更必须显示新增、删除、加强和降级的规则差异。
- 正文任务固定绑定生成时使用的规则包版本。
- 能力下线或证据失效时，必须检查受影响内容和蒸馏词。

### 12.3 回滚要求

- 保留上一版完整快照。
- 记录回滚原因、操作者、时间和受影响任务。
- 回滚不删除新版本证据，只将状态改为 `rolled_back`。

## 13. 最小验收标准

一份可用的产品规则包至少应做到：

1. 任何人能用一句话说清产品是谁。
2. 能区分已上线、条件成立、Beta 和规划中能力。
3. 每条强表述都能追溯到证据。
4. 缺证据时系统知道该降级还是拦截。
5. 能阻止产品混淆、绝对承诺和虚构案例。
6. 同一事实跨渠道保持一致。
7. 规则包可以确认、版本化和回滚。
8. 可以明确回答该产品能否进入月度内容生产池，以及可生产哪些内容类型和渠道。
