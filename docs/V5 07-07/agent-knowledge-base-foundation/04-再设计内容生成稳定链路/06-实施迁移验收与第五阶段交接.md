# 实施、迁移、验收与第五阶段交接

## 1. 目标

将阶段四接到 V5 月度内容矩阵、批量生成中心、异常稿处理和 PublishSchedule，同时兼容当前 V4 的 weekly-plan、today、drafts 和 publish 页面。

兼容原则：

- weekly-plan 降级为已批准月度矩阵的周视图。
- today 降级为当日排程、发布状态和失败接管视图。
- drafts 升级为异常稿处理和合格稿抽检。
- publish 升级为定时发布排程。
- 新 V5 写入不再创建独立周计划真源。

## 2. 当前可复用能力

- 平台内容类型和三项前置检查。
- 三产品标题体系和公众号自然表达标准。
- Brief 证据展示能力。
- Provider 调用和 Prompt 版本记录。
- 正文生成失败分类。
- A/B/C/D 草稿质量展示。
- 删除、AI 局部改写、保留风险和人工修改留痕。
- 角色权限和 AI 治理日志。

现有 Today 批量生成入口只作为迁移实现来源，V5 主入口必须迁到 Batch Generation Center。

## 3. 关键迁移

### 3.1 计划与任务

从 WeeklyPlan 和 confirmed ContentTask 迁移到：

- MonthlyPlan。
- MonthlyStrategyPackageVersion。
- ContentMatrixItem。
- approved matrix item execution snapshot。
- BatchGenerationRun。

旧 weeklyPlanId 和 taskId 作为 legacy 引用保留，不作为 V5 新写入主键。

### 3.2 EvidencePreview 与 Final EvidencePack

月度策略审核使用策略包自身的 `evidenceReadinessSummary`。策略批准并生成矩阵草稿后，EvidencePreview 用于月度矩阵和平台表达审核；矩阵批准后再按 matrixItemId 生成 Final EvidencePack。

### 3.3 GenerationSource

正式来源：

- ai_provider。
- manual_authored。

兼容或非正式来源：

- local_preview，只能用于预览和排查。

### 3.4 QA 拆分

现有 DraftQaResult 拆成：

- HardRuleValidationResult。
- SoftQualityEvaluationResult。
- DraftRoutingDecision。

普通合格稿进入抽检策略，异常稿进入人工队列。

### 3.5 发布

平台草稿只是某些平台的能力状态，不再是统一流程终点。

统一下游为 PublishSchedule：

- formal_publish：可以按已批准排程正式发布。
- draft_only：只能自动创建平台草稿。
- manual_only：生成交接包，由人发布并回填。
- pending_config：缺少真实平台配置。

## 5. 实施顺序

### Milestone 0：月度实体与兼容映射

建立 MonthlyPlan、ContentMatrixItem、BatchGenerationRun 和 PublishSchedule。旧周计划只读映射到月份和矩阵项。

### Milestone 1：月度策略、矩阵与 Evidence Gate

先接入月度策略审核；策略批准后接入 EvidencePreview、平台表达准备和月度矩阵审核。矩阵审核可以批准候选项并把 Preview 缺口项送入异常队列，但批准项仍需通过 Final Evidence Gate 才能进入批量生成。

### Milestone 2：矩阵项冻结和批量生成

矩阵批准后创建执行快照、Final EvidencePack 和 GenerationInput。Batch Generation Center 批量生成一个月正文。

### Milestone 3：硬规则

把夸大词、官网、产品词、Claim 对齐、能力状态、数字、隐私、人机角色和标题兑现升级为稳定 ruleId。

### Milestone 4：软评测、抽检与异常分流

接自然表达标准、结构评分、修改范围、抽检策略和异常队列。

### Milestone 5：PublishSchedule 与完整验收

只有 publish_schedule_ready 的 DraftVersion 才能创建排程。正式发布必须有人工前置确认和真实平台配置。

## 6. 固定验收样本

至少覆盖：

1. 月度矩阵未批准却尝试批量生成。
2. 矩阵项缺 active 规则包。
3. 标题有数字但无性能证据。
4. 规划能力被写成当前能力。
5. 产品能力串用。
6. 隐私政策与官网口径冲突。
7. 匿名案例泛化。
8. EvidencePack 外新增事实。
9. Provider 未配置产生 local_preview。
10. 局部改写引入新事实。
11. 标题修改导致旧 Pack 和 GenerationInput 失效。
12. 普通合格稿进入抽检策略。
13. 异常稿进入正确人工队列。
14. 平台只支持草稿却被误标为已发布。
15. 平台未配置时 PublishSchedule 返回 pending_config。

## 7. 工程验收

文档实现后至少运行：

    npm.cmd run typecheck
    npm.cmd run validate:structure
    npm.cmd run smoke:interactions
    node scripts/smoke-pages.mjs --base-url=http://127.0.0.1:3047
    node scripts/smoke-workflow.mjs --base-url=http://127.0.0.1:3047

新增契约测试：

- MonthlyPlan 未批准不能创建 BatchGenerationRun。
- Weekly View 和 Today Execution 不能改写冻结业务字段。
- Preview 和 Final Pack 不存在循环依赖。
- local_preview 不允许进入 PublishSchedule。
- Hard blocker 不被软分覆盖。
- 普通用户不能保留不可例外风险。
- 修改矩阵项会使旧 GenerationInput 过期。
- 发布排程绑定具体 matrixItemId 和 DraftVersion。
- formal_publish、draft_only、manual_only、pending_config 状态不会混淆。

## 9. 第五阶段前置条件

1. 问题代码稳定，不只存自由文本。
2. 人工修改和抽检原因使用标准分类。
3. 每个规则命中能定位文本和 Claim。
4. 草稿版本与最终发布版本可比较。
5. 评测模型版本和人工修正可追溯。
6. Demo、local_preview、ai_provider 和 manual_authored 明确区分。
7. 所有生产样本能追溯到月度矩阵项。

## 10. 阶段完成标准

阶段四完成时，系统从“周计划逐篇生成再人工通读”升级为：

    人工审核月度策略和月度矩阵
    → 矩阵项 Evidence Gate
    → 批量生成一个月正文
    → 独立硬规则裁判
    → 独立软质量评测
    → 合格稿抽检与异常精确分流
    → 合格稿进入 PublishSchedule
    → 数据进入评测和月度复盘闭环
