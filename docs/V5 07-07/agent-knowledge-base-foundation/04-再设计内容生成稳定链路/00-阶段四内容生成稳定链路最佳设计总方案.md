# 阶段四：内容生成稳定链路最佳设计总方案

## 1. 文档定位

本阶段负责把已批准月度内容矩阵项、平台表达方案、生效产品规则包和第三阶段证据资产，按 BatchGenerationRun 转化为可校验、可分流、可追溯的正文草稿。

它解决的不是“让模型自由写一篇文章”，而是：如何冻结生成输入、如何约束模型只能使用已批准事实、如何用独立规则引擎判断能否发布、如何把质量问题与事实风险分开，以及如何让人工只处理真正需要判断的异常。

阶段四的最终产物包括：

1. 冻结的 `GenerationInputSnapshot`。
2. 可重试、可复现的 `GenerationRun`。
3. 结构化硬规则结果 `HardRuleValidationResult`。
4. 独立软质量评测 `SoftQualityEvaluationResult`。
5. 合格稿与异常稿的 `DraftRoutingDecision`。
6. 人工修改、局部改写、风险保留与重新校验记录。

## 2. 核心判断

最佳链路不是：

```text
任务标题 + 若干 Chunk
-> 模型生成
-> 给文章打分
-> 人工通读修改
```

这会造成：

- 标题与证据互相倒推，缺少正式冻结点。
- 模型重新决定产品、受众、内容类型或核心判断。
- EvidencePack 外的常识被写成产品事实。
- 硬风险被质量平均分掩盖。
- Provider 缺失时本地模板稿被误认为生产正文。
- 所有问题最后都落到人工全文审核。

阶段四采用两次证据组装和四层裁判：

```text
月度目标与策略包草稿
-> 规则引擎校验与人工审核月度策略
-> ContentMatrixItem 草稿 + EvidencePreview
-> 平台表达准备与三项检查
-> 人工审核月度矩阵
-> 已批准矩阵项执行快照
-> Final EvidencePack
-> Final Evidence Gate
-> GenerationInputSnapshot
-> BatchGenerationRun 批量正文生成
-> Hard Rule Engine
-> Soft Quality Evaluator
-> Draft Router
```

## 3. 两次证据组装

### 3.1 `EvidencePreview`

发生在月度策略包和月度矩阵审核前，输入是：

- 产品与任务意图。
- 来源问题。
- 候选渠道。
- 主蒸馏词和知识库边界。
- active 规则包。

它回答：

- 哪些核心事实可以支撑本次选题。
- 哪些标题角度可证明。
- 哪些数字、案例和比较暂时不能使用。
- 当前更适合什么内容类型和受众。

EvidencePreview 只能辅助矩阵审核和平台表达准备，不能直接进入正式正文生成。月度策略审核使用策略包自身的 `evidenceReadinessSummary`。

### 3.2 `Final EvidencePack`

发生在月度策略和月度矩阵人工批准后。它根据已批准 ContentMatrixItem 的冻结执行快照生成最终 ClaimPlan 和证据槽位，回答：

- 最终标题中的每项承诺由哪些 Claim 证明。
- 正文允许使用哪些事实和原文。
- 哪些条件必须写入。
- 哪些主张禁止使用。
- 是否允许生成、需要降级、补资料、人工确认或阻断。

通过两次组装解除“先有标题还是先有 EvidencePack”的循环依赖。

## 4. 阶段边界

### 4.1 本阶段负责

1. 月度策略与矩阵审核后的矩阵项执行快照。
2. Final EvidencePack 生成请求与结果校验。
3. GenerationInput 版本化和幂等控制。
4. Provider、Model、Prompt 和超时重试编排。
5. 按 BatchGenerationRun 批量生成一个月正文，并支持指定范围重新生成。
6. 硬规则校验和软质量评测。
7. 合格稿、补资料、人工修改、重新生成、渠道重写和配置诊断分流。
8. 人工删除、局部改写、风险保留和版本差异留痕。
9. 为第五阶段输出样本、规则命中、人工动作和最终结果。

### 4.2 本阶段不负责

1. 不重新识别产品实体和 ProductClaim。
2. 不裁决来源冲突或修改证据等级。
3. 不重新决定产品、渠道、标题、内容类型、主要受众和主蒸馏词。
4. 不绕过第三阶段重新全库检索。
5. 不自动批准产品规则包或修改 active 版本。
6. 不直接执行发布；合格稿只交给 PublishSchedule，正式发布由已批准排程的发布执行服务完成。
7. 不负责 Workflow Agent 跨任务调度。

## 5. 正式生产链路

```text
MonthlyPlan 与月度策略包
-> MonthlyStrategyApproval
-> ContentMatrixItem 草稿
-> EvidencePreview
-> PlatformExpressionPlan
-> PlatformExpressionPrecheck
-> MonthlyMatrixApproval
-> ApprovedMatrixItemExecutionSnapshot
-> Final EvidencePack
-> FinalEvidenceGate
-> GenerationReadinessCheck
-> GenerationInputSnapshot
-> BatchGenerationRun / GenerationRun
-> HardRuleValidation
-> SoftQualityEvaluation
-> DraftRoutingDecision
-> Exception Handling / Sampling
-> PublishSchedule
```

## 6. 三类输入状态不能混淆

### 业务任务状态

描述标题、平台内容类型、受众是否已确认。

### 证据状态

描述 EvidencePreview/Pack 是否充分、有冲突或需要降级。

### 模型运行状态

描述 Provider 是否配置、调用是否成功、结构化输出是否有效。

模型成功不能替代证据通过；证据充分也不代表 Provider 已可用；任务人工确认也不允许绕过硬规则。

## 7. 四层裁判

### 7.1 生成准备校验

判断是否值得调用模型，避免缺输入时浪费调用。

### 7.2 生成器

只负责在冻结输入内组织正文，不负责决定事实可用性。

### 7.3 硬规则引擎

确定是否存在不可发布风险，不受软质量分影响。

### 7.4 软质量评测

判断观点、结构、证据展开、自然度和渠道适配，不能覆盖硬规则结果。

## 8. 本地 fallback 边界

当前 V4 在 Provider 缺失或失败时会生成本地规则稿。V5 需要改变其业务语义：

- 可以生成 `local_preview`，用于界面演示、结构预览和排查。
- 状态必须为 `pending_config` 或 `provider_failed`。
- 不得标记为 AI 正式生成成功。
- 不得进入抽检通过状态或 PublishSchedule。
- 不得通过人工简单勾选绕过 Provider 与证据要求。

如果业务确需人工从零撰写，应创建 `manual_authored` 来源，进入完整硬规则与质量校验，而不是把本地模板稿伪装成生产生成结果。

## 9. 角色边界

- 内容发布人员：抽检合格稿、处理异常稿、编辑正文、完成发布前置确认和失败接管。
- 内容增长人员：维护平台内容类型、标题体系和质量标准。
- 知识库维护：处理证据缺口和来源问题。
- 工作台运营：处理 Provider、Prompt、规则运行和异常趋势。
- 产品/技术/安全/法务 Owner：处理高风险事实和例外。
- 开发管理员：查看底层运行、重试和技术日志，不替代业务确认。

## 11. 阶段完成标准

1. 平台表达准备使用 EvidencePreview，最终正文只使用 Final EvidencePack。
2. GenerationInput 冻结后模型不能重新决定业务字段。
3. 正文中的产品强事实可以逐条追溯到 EvidenceItem/Claim。
4. Provider 缺失或失败不会产生可发布伪结果。
5. 硬风险不受质量评分影响，阻断漏判为 0。
6. 软质量问题与事实/证据问题分别分流。
7. 合格稿和异常稿进入明确队列，人工只处理例外。
8. 所有正式生成记录都绑定 monthlyPlanId、matrixVersionId、matrixItemId 和 batchGenerationRunId。
9. 周视图和当日执行不能成为正文批量生成入口。
10. 合格稿只进入 PublishSchedule，发布前置确认与真实平台能力由后续调度层校验。
11. 所有生成、校验、修改、保留和重新生成有版本与原因。
12. 第五阶段可以使用结构化运行数据构建正向、边界和 Badcase 评测集。
