## 1. 结论

当前可以进入知识库 V5 的真实开发，但不能进入真实生产或自动发布。

允许开始的是：

1. 月度 Schema、稳定 ID、迁移 dry-run 和差异报告。
2. 资料接入、敏感检查、SourceAsset、ProductClaim 和冲突治理的真实垂直切片。
3. 规则包人工激活、MonthlyProductionReadiness 和生产池准入。
4. 以同一领域策略为入口的 RAG、矩阵、生成、质检和发布后续开发。

当前禁止的是：

1. 把四个品牌/产品规则草案直接标记为 active。
2. 把历史 1297 个 Chunk 直接视为 V5 生产证据。
3. 在月度实体和 Final Evidence Gate 未建立前让 Agent 操作旧周计划生成正文。
4. 把 local_preview、draft_only、manual_only 或 pending_config 记为真实生成或正式发布成功。

## 4. 可执行规则映射

以下规则由 `evaluateV5KnowledgeWorkflow(target, context)` 统一执行。页面、Agent、Scheduler 和脚本后续必须调用同一领域服务，不得建立旁路写接口。

| 规则 | 代码级阻断 | 原理 | 对使用者的影响 |
| --- | --- | --- | --- |
| V5-KB-001 | 敏感检查未完成或命中时禁止事实抽取 | G0 先于解析和模型调用 | 隐私和凭证不会进入外部模型 |
| V5-KB-002 | 来源未解析、未归属或不可定位时禁止抽取 | 先来源、后事实 | 每条事实可回到原文 |
| V5-KB-003 | Agent 禁止激活规则包 | 核心判断权归人 | AI 建议不会自动变生产规则 |
| V5-KB-004 | 未解决阻断冲突禁止继续 | 默认保守 | 冲突不会静默污染正文 |
| V5-KB-005 | 非 active 或 readiness=false 禁止入池 | 月度准入独立于上传成功 | 待确认产品不会被误生产 |
| V5-KB-006 | 缺 monthlyPlanId 等稳定 ID 禁止写入 | 月度矩阵是真源 | 全链路可追溯 |
| V5-KB-007 | 策略未批准或无 EvidencePreview 禁止矩阵审批 | 先策略、后矩阵 | 审核对象不循环依赖 |
| V5-KB-008 | 平台表达检查或人工确认未通过时禁止冻结 | 标题和证据边界前置 | 模型不能临时改业务判断 |
| V5-KB-009 | 无 Final EvidencePack 或 Gate 未通过时禁止生成 | Preview 不授予生成权 | 缺证据项不消耗模型产正文 |
| V5-KB-010 | 缺真实配置返回 pending_config | 不伪造能力 | 用户看到真实阻塞和下一步 |
| V5-KB-011 | local_preview 或质检失败禁止排程 | 生成与发布分层 | 伪生产稿不进入发布队列 |
| V5-KB-012 | 非 formal_publish、未审批或未排程禁止正式发布 | 发布能力与状态分离 | 草稿箱和人工接管不算发布 |
| V5-KB-013 | 无平台成功证明或 URL 禁止写 published | 结果以外部事实为准 | 发布率只统计真实内容 |
| V5-KB-014 | 缺幂等键、版本或审计字段禁止写入 | 所有入口共用写约束 | 重试不会制造重复副作用 |
| V5-KB-015 | 复盘把非发布状态计为成功时阻断 | 指标只读真实结果 | 下月策略不被虚假成功率误导 |

## 5. 最简实施顺序

### 波次 1：数据与策略骨架

1. 建立 MonthlyPlan、ContentMatrixVersion、ContentMatrixItem 和稳定 ID。
2. 建立迁移适配器，只读旧 weekly 字段并输出 dry-run 差异。
3. 将本策略模块接到统一领域写服务，不直接改 V4 页面行为。

### 波次 2：一个真实知识库垂直切片

1. 选一个产品的 10 至 20 份代表资料。
2. 跑 G0 至 G6：安全、解析、实体、事实、冲突、规则包和月度准入。
3. 产出唯一 RAG Ingestion Manifest，不全量导入历史 Chunk。

### 波次 3：月度矩阵到生成

1. 先策略人工批准，再创建矩阵草稿和 EvidencePreview。
2. 矩阵人工批准后生成 Final EvidencePack。
3. 只有 Gate 通过项进入 BatchGenerationRun。

### 波次 4：发布与复盘

1. 质检通过稿进入 PublishSchedule。
2. 分清 formal_publish、draft_only、manual_only 和 pending_config。
3. 用真实平台结果、URL 和渠道数据生成 MonthlyReview。

## 6. 本次实现边界

本次已建立 V5 领域策略和回归测试，但没有把现有 V4 WeeklyPlan 静默改造成 V5。

原因：月度表、迁移适配器和生产数据尚未建立。此时直接修改旧入口，只会让界面名称变成 V5，底层仍写 V4 真源，长期维护风险更高。

替代简化方案是先让策略模块独立可测，再在波次 1 的统一领域服务中接入。这样 V4 兼容读取和 V5 新写入可以明确分开。
