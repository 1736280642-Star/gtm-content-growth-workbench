# V5 波次一：绿色月度数据底座与 V4 表删除记录

实施日期：2026-07-14  
实施分支：`feat/workbench-v3`  
实施范围：V5 领域契约、数据库 Schema 和后端 cutover

## 1. 结论

波次一已调整为绿色 V5 建设，不再兼容或迁移 V4 周度业务数据。

真实 MySQL cutover 已完成：

- 7 张原生 V5 表已创建。
- 4 张 V4 周度关系表已永久删除。
- V4 WeeklyPlan 和 ContentTask 数据未迁移到 V5。
- 所有 legacy 映射字段、迁移映射表和 dry-run API 已移除。
- 两个 migration 的 checksum 已写入 `workbench_schema_migration`。

## 2. 为什么撤销 V4 兼容

原方案保留兼容，是基于“旧周度数据可能需要审计和迁移”的保守假设。用户明确旧周度数据没有业务价值后，继续兼容会产生以下负担：

1. MonthlyPlan 和 WeeklyPlan 形成双真源。
2. 每个新字段都要维护旧字段映射。
3. 旧 confirmed 状态容易被误认为 V5 人工批准。
4. 重复迁移、差异报告和 legacy ID 增加长期维护成本。
5. 前后端需要长期理解两套计划语义。

因此最简路径是从空的 V5 MonthlyPlan 开始，旧周度关系表直接删除。

## 3. 最终 V5 数据表

| 表 | 作用 |
| --- | --- |
| `monthly_plan` | 月度目标和计划真源 |
| `monthly_strategy_package_version` | 不可变月度策略包版本 |
| `content_matrix_version` | 不可变月度矩阵版本 |
| `content_matrix_item` | 月度内容任务真源 |
| `monthly_production_readiness` | 产品月度生产准备度 |
| `production_pool_entry` | 产品进入指定月份生产池的记录 |
| `artifact_reference` | 跨对象、版本和阶段追溯 |

这些表不包含 `legacyWeeklyPlanId`、`legacyTaskId`、旧规则包引用或 V4 迁移快照。

## 4. 已删除的表

以下关系表已从基础 `database/schema.sql` 移除，并通过 destructive migration 从真实 MySQL 删除：

1. `weekly_plan`
2. `content_task`
3. `article_draft`
4. `publish_record`

上一版短暂设计的兼容表也已删除：

- `v5_migration_run`
- `v5_migration_item_map`

知识库、博客、GEO、Workspace、审计和通用状态快照表未删除。

## 5. 安全执行边界

删除 migration 必须显式执行：

```powershell
node scripts/init-v5-monthly-schema.mjs --confirm-drop-v4
```

不带确认参数时，执行器返回 `confirmation_required`。

查看计划不会连接或修改数据库：

```powershell
node scripts/init-v5-monthly-schema.mjs --plan
```

核对真实数据库：

```powershell
node scripts/check-v5-schema-cutover.mjs
```

## 7. 分环节自检

### 环节 1：移除兼容代码

- 删除 V4 到 V5 dry-run 规划器。
- 删除 legacy 数据契约。
- 删除迁移 CLI 和只读迁移 API。
- 原生 V5 TypeScript 契约不包含 V4、WeeklyPlan 或 legacy 字段。

结论：业务层不再维护旧数据映射。

### 环节 2：Schema cutover

- 基础 schema 不再创建 4 张 V4 周度表。
- V5 foundation 只创建 7 张原生 V5 表。
- destructive migration 只删除明确列出的旧表。

结论：数据库不再保留周度关系模型。

### 环节 3：删除确认和真实执行

- 不带确认参数会拒绝执行。
- `--plan` 会标记 destructive migration。
- 带 `--confirm-drop-v4` 后真实执行成功。
- 只读核对确认旧表为 0、V5 表为 7。

结论：删除操作有显式边界且已完成。

### 环节 4：验证

- Schema cutover 测试：6/6 通过。
- TypeScript：通过。
- 敏感凭证检查：0 命中。
- 其余全仓验证以最终任务输出为准。

## 8. 当前剩余边界

关系表已经删除，但当前工作台仍使用通用 `workbench_state_snapshot` 或本地 JSON 维持现有页面运行，其中可能包含 V4 周度字段。

本轮没有修改 `src/lib/types.ts`、`src/lib/workbench-store.ts` 或前端页面，因为这些文件属于并行分支高冲突区。前端分支合并并接入 MonthlyPlan 后，后端下一波需要：

1. 建立 V5 Monthly Repository。
2. 让 API 直接读写 7 张 V5 表。
3. 停止把 WeeklyPlan、ContentTask、ArticleDraft 和 PublishRecord 写入通用状态快照。
4. 验证新入口稳定后，再清理状态快照里的 V4 JSON 字段。

这不是保留 V4 兼容，而是避免在新运行入口尚未完成时让当前工作台立即不可用。
