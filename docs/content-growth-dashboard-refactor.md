# 工作台内容增长与 GEO 监控改造梳理

## 1. 文档定位

本文档用于梳理工作台下一阶段的产品改造边界，不涉及代码实现。

本次讨论的核心目标不是继续堆页面，而是把工作台从“内容生产流程工具”升级为“公司自动化内容生产与数据监控闭环”。这个闭环需要同时服务两类目标：

1. 给人看：渠道内容要能被真实用户理解、点击、收藏、转发。
2. 给 AI 看：官网、博客和渠道内容要围绕 AI 认知节点建立长期权重，提高被模型理解、引用和关联到 JOTO / 唯客的概率。

## 2. 可信等级是否保留

### 2.1 建议结论

可信等级不建议继续作为用户手动设置项保留。

原因是：当前系统里的“可信等级”需要用户判断，但用户在导入资料时真正关心的是“这份资料从哪里来、能不能被用于生成、是否需要人工确认”。如果让用户手动选择最高、高、中、参考，反而会增加认知负担，并且容易形成伪精确判断。

### 2.2 替代方式

建议删掉用户可见的“可信等级”，改成系统内部根据来源类型和审核状态判断调用权限。

可保留的字段应是：

| 字段          | 用户是否需要看到 | 用途                            |
| ----------- | -------- | ----------------------------- |
| 知识库名称       | 需要       | 用户识别知识库                       |
| 知识库类型       | 需要       | 品牌事实、产品资料、官网博客、渠道历史、竞品参考、外部来源 |
| 来源 URL / 文件 | 需要       | 追溯来源                          |
| 调用范围        | 需要       | 决定哪些流程可以用                     |
| 状态          | 需要       | 启用 / 停用                       |
| 审核状态        | 可选       | 待确认 / 已确认                     |
| 来源类型        | 可自动生成    | URL、Markdown、Docx、手动输入、自动抓取   |

系统内部可以根据规则控制：

1. 品牌事实、产品资料、官网博客：可进入内容生成和事实引用。
2. 渠道历史：可用于风格、选题、渠道表达参考。
3. 竞品参考：只能用于对比、差异化和市场分析。
4. 外部来源：默认不直接作为事实来源，除非人工确认。
5. 待确认资料：可以预览和手动引用，但不自动进入批量生成。

### 2.3 业务效果

删除可信等级后，知识库页面会更接近真实工作流：

```text
导入资料 -> 设置名称和类型 -> 设置调用范围 -> 预览内容 -> 启用 / 停用
```

用户不需要先判断“可信等级”，系统也不会因为一个主观字段误导后续生成。

## 3. 知识库导入：标准导入 vs 切片与 Chunk 预览

### 3.1 标准化导入是什么

标准化导入指的是：

```text
URL / Markdown / Docx / 手动输入
-> 解析成纯文本或 Markdown
-> 展示预览
-> 设置知识库名称、类型、调用范围
-> 保存为知识库
-> 后续生成文章时作为上下文资料使用
```

它解决的是“资料能不能进系统、能不能被管理、能不能被后续流程调用”。

### 3.2 切片与 Chunk 预览是什么

切片与 Chunk 预览指的是：

```text
导入资料
-> 清洗正文
-> 按标题、段落、语义或模型规划拆成多个 chunk
-> 每个 chunk 有标题、来源、段落路径、token 数、内容摘要
-> 用户可以预览每个 chunk
-> 后续检索时按 chunk 命中，而不是整篇资料塞给模型
```

它解决的是“模型在生成文章时，能不能精准拿到相关事实，而不是拿到一大坨无关资料”。

### 3.3 两种方案区别

| 维度 | 标准化导入 | 切片与 Chunk 预览 |
|---|---|---|
| 开发成本 | 中 | 高 |
| 上线速度 | 快 | 慢 |
| 用户理解成本 | 低 | 中 |
| 对短文档支持 | 足够 | 更好但不一定必要 |
| 对长文档支持 | 容易上下文过长 | 更稳定 |
| 对精准引用 | 一般 | 明显更好 |
| 对后续 RAG | 只是前置 | 是核心基础 |
| 是否需要向量化 | 不必须 | 后续最好接入 |

### 3.4 对 AI 生成文章的影响

如果只做标准化导入，生成文章时通常是：

```text
任务信息 + 若干知识库摘要 / 全文片段 -> 模型生成
```

优点是实现快，适合当前先跑通真实业务流。缺点是文档一多，模型容易：

1. 读到无关信息。
2. 漏掉关键事实。
3. 只引用知识库名称，不真正使用知识内容。
4. 长内容超出上下文，导致事实链路不稳定。

如果做切片与 Chunk 预览，生成文章时会变成：

```text
任务信息 + 蒸馏词 + 用户问题
-> 检索相关 chunk
-> 只把相关事实片段交给模型
-> 生成文章
```

效果会更稳定：

1. 文章更容易引用准确事实。
2. 官网链接和产品信息更不容易丢。
3. 蒸馏词、品牌词、产品卖点之间更容易建立因果关系。
4. 后续可以复盘“哪段知识支撑了这篇文章”。

### 3.5 建议的落地边界

不建议第一步就做完整 RAG、向量库和复杂语义检索。

更合适的路线是分三层：

| 阶段  | 功能                      | 目的                   |
| --- | ----------------------- | -------------------- |
| V1  | 标准化导入 + 内容预览 + 来源记录     | 先让真实资料进入系统           |
| V2  | 规则切片 + Chunk 预览         | 让长文档可控，让用户知道模型会用哪些片段 |
| V3  | 语义检索 / 向量化 / 自动推荐 chunk | 让内容生成长期稳定            |

## 4. 自动化抓取站点与知识库更新

### 4.1 需求理解

你一开始要的自动化能力，本质不是“自动爬网页”这么简单，而是：

```text
定期抓取指定站点
-> 发现新增 / 更新内容
-> 解析正文
-> 和旧知识库比对
-> 生成待导入或待更新记录
-> 用户确认后进入知识库
-> 需要时触发切片更新
```

### 4.2 有效衔接方式

自动化抓取不应该绕过知识库导入流程，而应该复用同一条导入管线。

推荐结构：

```text
手动导入 URL / 文件
        |
        v
统一解析管线 -> 预览 -> 设置知识库信息 -> 保存 -> 启用
        ^
        |
自动抓取站点 -> 发现新内容 / 内容变化
```

也就是说，自动化只是“把 URL 和原始内容送进导入队列”，后面的解析、预览、切片、启用逻辑都应共用。

### 4.3 自动化可设置项

自动化配置应支持：

| 配置项 | 说明 |
|---|---|
| 抓取名称 | 例如 JOTO 官网博客、产品文档、竞品博客 |
| 抓取站点 | sitemap、RSS、列表页、固定 URL |
| 频率 | 每天、每周、每月、手动 |
| 内容归属 | 导入到哪个知识库类型 |
| 调用范围 | 进入内容生成、博客监控、GEO 测试、仅候选 |
| 更新策略 | 自动进入待确认、自动更新已有关联、仅生成差异报告 |
| 停用按钮 | 停止后不再抓取，但保留历史记录 |

### 4.4 业务效果

加上自动化抓取后，知识库会从“人工录入资料”变成“持续更新的内容资产池”。

长期效果：

1. 官网新增文章后可以自动进入博客监控。
2. 产品资料更新后可以提示相关内容需要重写。
3. 竞品内容更新后可以进入对比分析和候选池。
4. 周报可以看到“本周新增知识资产”和“本周内容资产变化”。

### 5.1 页面目标

GEO 测试页不应只是执行测试和展示矩阵，而应成为诊断复盘台。

用户打开页面后，应先看到：

1. 本轮测试结果是否健康。
2. 主要问题是什么。
3. 问题原因可能是什么。
4. 下一步建议动作是什么。
5. Prompt 组是什么、能不能编辑。

## 6. 首页数据看板改造

### 6.1 当前首页问题

当前首页已经有一些看板和下一步动作，但还偏“任务清单”。用户打开首页时，应该先看到本周从计划到发布的整体进度，再看到工作台的流程结构。

### 6.2 首页应展示的核心指标

首页顶部建议展示本周开始到当天的四个主指标：

| 指标 | 定义 |
|---|---|
| 计划数量 | 本周计划内应完成的内容任务数 |
| 生成数量 | 已生成草稿或进入终稿确认的数量 |
| 已发布数量 | 已标记发布或已回填 URL 的数量 |
| 待回填数量 | 已发布但尚未回填正式 URL 的数量 |

辅助指标：

1. 待终稿确认。
2. 发布队列待处理。
3. GEO 待处理。
4. 博客待优化。
5. AI Bot PV。

## 7. 周度复盘改造与低保真图

### 7.1 当前“生成下周计划草稿”的来源

当前逻辑里，周报页的“生成下周计划草稿”来自 `getWeeklyReport()` 返回的 `nextWeekSuggestions`。

再由 `createNextWeeklyPlanFromReport()` 把这些建议拼进新任务标题里，生成下一周任务。

现有链路大致是：

```text
周报建议 nextWeekSuggestions
-> 按默认发布天数 / 每日数量 / 渠道 / 产品
-> 循环生成任务
-> 标题 = 本地规则标题 + 周报建议
-> 覆盖当前周计划任务、草稿、发布队列
```

这个链路现在的问题是：

1. 生成前没有预览。
2. 用户不知道会生成哪些选题。
3. 不知道这些选题命中了哪些蒸馏词。
4. 不知道它们解决了上周哪些问题。
5. 直接在周报页生成，会让“复盘”和“计划管理”职责混在一起。

### 7.2 建议结论

更建议把周报页的按钮从“生成下周计划草稿”改成“进入下周计划预览”。

周报页负责回答：

```text
上周发生了什么？
哪些问题需要下周处理？
```

周计划页负责回答：

```text
下周具体生成哪些任务？
每个任务选题是什么？
命中哪些蒸馏词？
解决上周哪个问题？
用户是否确认生成？
```

### 7.3 周度复盘低保真图

## 8. 博客监控是否适合加入 geo-optimizer-skill

### 8.1 当前博客监控链路

当前博客监控已有链路：

```text
导入博客数据 / sitemap / JSON / CSV
-> 博客列表
-> 判断收录状态、SEO 问题数、GEO 结果
-> 诊断
-> 加入候选池
-> 生成渠道补强任务
```

当前字段主要是：

1. 标题。
2. URL。
3. 收录状态。
4. SEO 问题数量。
5. GEO 命中 / 未命中 / 部分命中。
6. 日志可信度。
7. 候选池状态。

它已经有“监控 -> 诊断 -> 入候选池 -> 进入内容生产”的业务链路，但诊断指标偏粗。

### 8.2 geo-optimizer-skill 检测的是什么

geo-optimizer-skill 更偏“页面是否适合被 AI 引用”的审计。

它关注：

1. AI crawler 是否能访问。
2. 是否有 `llms.txt`。
3. schema 是否完整。
4. meta、canonical、Open Graph 是否完整。
5. 页面内容结构是否清晰。
6. 是否有数据、引用、来源。
7. 是否有负面信号。
8. RAG chunk readiness。
9. trust stack。
10. citation readiness。

### 8.3 是否会更稳定

它会让博客监控的“基础页面审计”更稳定，但不会替代 GEO 测试。

原因是：

1. geo-optimizer-skill 是确定性检查，适合判断页面是否具备被引用的基础条件。
2. GEO 测试是模型回答测试，适合判断 AI 实际是否会提到 JOTO / 唯客 / 官网。
3. 两者目的不同，不能互相替代。

### 8.4 是否适合加入

从真实业务目标看，适合加入，但应放在博客监控，而不是 GEO 测试的核心执行里。

公司长期做自动化内容生产和数据监控，需要同时回答两件事：

1. 我的内容有没有持续生产？
2. 我的站点和内容有没有越来越适合被 AI 理解和引用？

geo-optimizer-skill 正好补第二件事。

### 8.5 推荐接入方式

推荐在博客监控里增加“GEO 页面审计”模块：

```text
博客 URL
-> 页面抓取
-> geo optimizer 审计
-> 输出页面引用准备度
-> 合并当前 SEO / GEO 结果
-> 决定是否入候选池
```

新增指标可以是：

| 指标 | 用途 |
|---|---|
| GEO 页面分 | 总体页面可引用性 |
| AI crawler 可访问 | 判断技术阻断 |
| llms / schema 状态 | 判断结构化基础 |
| citation readiness | 判断是否适合作为 AI 回答来源 |
| negative signals | 判断是否有堆砌、薄内容、隐藏指令 |
| content evidence score | 判断是否有数据、来源、明确结论 |
| chunk readiness | 判断是否适合进入知识库切片 |

### 8.6 业务效果

加入后，博客监控会从：

```text
这篇博客有没有问题？
```

升级为：

```text
这篇博客能不能成为 AI 回答中的可信信源？
如果不能，是内容问题、结构问题、技术问题，还是引用链路问题？
```

这对长期内容增长更有价值。

### 8.7 边界

第一版不要追求自动修复所有 robots、llms、schema。

第一版只做：

1. 读取 URL。
2. 输出审计结果。
3. 保存核心指标。
4. 给出候选池建议。

自动生成 `llms.txt`、schema 注入、批量修复可以放到后续版本。

## 9. 内容生产链路优化

### 9.1 需求理解

你要的内容生产标准不是单一 SEO，也不是单一 GEO。

一篇内容要同时满足四类要求：

1. 命中蒸馏词：让 AI 把文章和关键认知节点绑定。
2. 命中品牌词：让 JOTO / 唯客被稳定提及。
3. 命中官网链接和产品推广重点：让内容有可信来源和业务转化方向。
4. 符合渠道风格：让内容不是一篇文章复制到所有平台，而是按渠道重新组织表达。

### 9.2 通用规则层

不管采用哪条生产链路，都应有一组基础规则。

#### 规则 1：每篇文章必须有一个主蒸馏词

作用：防止内容只围绕标题和关键词发散。

示例：

```text
主蒸馏词：AI 输出安全
用户问题：Dify 应用上线后，如何避免模型输出不合规内容？
产品关联：唯客 AI 护栏
```

#### 规则 2：每篇文章必须有一个用户真实问题

作用：防止内容变成品牌介绍或概念堆砌。

好的任务不是“写 AI 护栏”，而是：

```text
企业把 Dify 应用接入客服场景后，怎么避免模型输出错误承诺？
```

#### 规则 3：品牌词必须和蒸馏词建立因果关系

不要只写：

```text
JOTO 提供相关服务。
```

要写成：

```text
当企业把 AI 应用从 Demo 推到真实业务，输出安全会从提示词问题变成治理问题。JOTO 的唯客 AI 护栏适合放在这个环节，帮助团队在发布前识别高风险输出。
```

#### 规则 4：官网链接必须有引用理由

官网链接不是为了塞链接，而是为了证明：

1. 产品能力来自哪里。
2. 某个事实是否可追溯。
3. 用户下一步应该看哪类官方资料。

#### 规则 5：不同渠道使用不同内容选材

| 渠道 | 内容重点 |
|---|---|
| 公众号 | 观点、认知解释、业务判断、案例复盘 |
| CSDN | 技术拆解、流程说明、工程实践、配置步骤 |
| 掘金 | 开发者视角、架构设计、实践经验、踩坑 |
| 知乎 | 问题回答、选型对比、利弊判断 |
| 头条 | 更直接的问题、结论和案例 |

## 10. 几种可选内容生产链路

### 10.1 链路 A：蒸馏词优先链路

#### 流程

```text
蒸馏词库
-> 选择主蒸馏词
-> 生成用户问题
-> 匹配产品词和官网证据
-> 分配渠道
-> 生成文章
-> GEO 结构质检
-> 发布
-> GEO 测试复盘
```

#### 适用场景

适合长期建设某个 AI 认知节点，比如：

1. 企业级 AI 应用治理。
2. AI 输出安全。
3. Dify 企业级交付。
4. AI 应用可信交付。

#### 业务效果

优点：

1. 最有利于长期 GEO 权重建设。
2. 内容矩阵更稳定。
3. 周报能按蒸馏词复盘。

风险：

1. 如果蒸馏词库质量不高，会导致选题方向偏。
2. 需要定期用多模型回答验证蒸馏词是否真实存在。

#### 推荐规则

1. 每个高优先级主蒸馏词至少产出 5 类内容：认知解释、FAQ、技术拆解、对比判断、案例复盘。
2. 每个主蒸馏词至少绑定 1 篇官网博客作为可信来源。
3. 每篇内容只设 1 个主蒸馏词，最多 2-3 个辅助蒸馏词。
4. 周报按蒸馏词覆盖和 GEO 命中情况决定下周继续写、补强写、暂停写。

### 10.2 链路 B：问题优先链路

#### 流程

```text
渠道反馈 / GEO 未命中 / 用户问题
-> 归类为具体问题
-> 匹配蒸馏词
-> 匹配品牌词和产品卖点
-> 选择渠道表达方式
-> 生成文章
-> 发布后看渠道数据和 GEO 测试
```

#### 适用场景

适合处理真实反馈和短期增长问题，比如：

1. 某个知乎问题适合回答。
2. GEO 测试发现模型没有提到 JOTO。
3. 某篇博客有 SEO / GEO 问题。
4. 某个渠道数据表现好，需要延展。

#### 业务效果

优点：

1. 更贴近真实用户问题。
2. 更容易获得渠道阅读和互动。
3. 能快速修复当前内容缺口。

风险：

1. 容易被短期问题牵着走。
2. 如果不绑定蒸馏词，容易回到传统关键词内容。

#### 推荐规则

1. 每个问题必须映射到一个主蒸馏词。
2. 如果找不到蒸馏词，先进入“蒸馏词候选”，不要直接批量生产。
3. 问题来源必须记录：渠道数据、GEO 测试、博客监控、人工输入。
4. 生成后必须回填“这个内容解决了哪个问题”。

### 10.3 链路 C：渠道优先链路

#### 流程

```text
选择渠道
-> 选择该渠道适合的内容类型
-> 匹配蒸馏词和产品重点
-> 生成渠道化选题
-> 生成渠道化内容
-> 发布
-> 导入渠道数据
-> 复盘渠道表现
```

#### 适用场景

适合你要稳定运营 CSDN、掘金、知乎、公众号这些渠道时使用。

#### 业务效果

优点：

1. 最适合保证发布节奏。
2. 渠道风格更稳定。
3. 适合团队化、标准化执行。

风险：

1. 容易为了渠道更新而生产低战略价值内容。
2. 如果没有蒸馏词约束，内容会散。

#### 推荐规则

1. 每个渠道都有自己的内容类型配比。
2. 每篇渠道内容必须绑定主蒸馏词。
3. 同一主蒸馏词可以跨渠道改写，但不能简单复制。
4. 渠道数据进入周报后，要反哺蒸馏词优先级。

### 10.4 链路 D：官网信源优先链路

#### 流程

```text
官网博客 / 产品文档 / 品牌事实库
-> 提取可信事实
-> 绑定蒸馏词
-> 生成渠道补强内容
-> 发布并回链官网
-> GEO 测试观察官网引用率
```

#### 适用场景

适合解决“AI 提到了品牌但不引用官网”的问题。

#### 业务效果

优点：

1. 最有利于提高官网引用率。
2. 能把渠道内容和官网资产连接起来。
3. 更适合长期信源建设。

风险：

1. 如果官网内容本身薄弱，渠道内容会缺少事实支撑。
2. 需要博客监控和 geo optimizer 审计协同。

#### 推荐规则

1. 每篇渠道内容至少有一个官网事实或官网链接目标。
2. 如果官网没有可引用事实，先生成官网博客候选，而不是直接写渠道推广文。
3. 官网内容优先做结构化：标题清晰、FAQ、How-to、数据、来源、产品边界。
4. 周报单独追踪官网引用率。

### 10.5 链路 E：监控反哺链路

#### 流程

```text
博客监控 / GEO 测试 / 渠道数据 / AI Bot 日志
-> 识别问题
-> 入候选池
-> 匹配蒸馏词
-> 生成补强任务
-> 发布
-> 再监控
```

#### 适用场景

适合形成长期闭环。

#### 业务效果

优点：

1. 能让内容生产不再靠拍脑袋。
2. 每篇补强内容都有来源问题。
3. 适合长期管理内容增长。

风险：

1. 需要数据质量稳定。
2. 初期没有真实日志时，部分指标只能用导入数据或人工判断。

#### 推荐规则

1. 候选池必须记录来源：博客问题、GEO 未命中、竞品占位、渠道高表现、人工判断。
2. 候选任务必须记录“要优化上一周哪个问题”。
3. 没有明确优化目标的候选，不自动进入周计划。
4. 周报要追踪候选任务发布后的效果。

## 11. 推荐组合

不建议只选择一种链路。

更适合当前工作台的组合是：

```text
蒸馏词优先链路：负责长期权重
问题优先链路：负责短期缺口
渠道优先链路：负责稳定发布
官网信源优先链路：负责可信引用
监控反哺链路：负责复盘闭环
```

对应到业务节奏：

| 周期 | 主链路 | 目标 |
|---|---|---|
| 每天 | 渠道优先 + 问题优先 | 保持内容生产和发布 |
| 每周 | 监控反哺 + 周报复盘 | 找出下周补强方向 |
| 每月 | 蒸馏词优先 | 检查内容矩阵是否成型 |
| 持续 | 官网信源优先 | 提高官网引用率和可信来源权重 |

## 12. 下一阶段建议

### 12.1 P0

1. 删除前端“可信等级”设置项，改为来源类型、调用范围、启用状态。
2. 知识库导入先做标准化导入，不急着完整向量化。
3. GEO 测试改成诊断复盘台，并增加 Prompt 组管理抽屉。
4. 首页改成数据看板，展示本周计划、生成、发布、待回填。
5. 周报按钮改为“进入下周计划预览”，不要在周报页直接生成并覆盖计划。

### 12.2 P1

1. 知识库增加规则切片和 Chunk 预览。
2. 周计划页增加“下周计划生成预览”。
3. 博客监控接入 geo optimizer 审计结果。
4. 内容任务增加蒸馏词、用户问题、官网证据、产品推广重点。

### 12.3 P2

1. 自动化抓取站点，支持频率和抓取范围配置。
2. 语义检索和向量化。
3. 蒸馏词矩阵复盘。
4. 多模型交叉验证蒸馏词。

## 13. 总结

这次改造的关键不是增加“更复杂的 AI 能力”，而是把每个流程的业务责任划清楚。

知识库负责提供可信输入；博客监控负责判断官网内容是否适合被引用；GEO 测试负责判断模型是否真的想到 JOTO；周报负责识别问题；周计划负责把问题变成可执行任务；内容生成负责同时满足人类阅读、AI 认知节点、品牌关联、官网引用和渠道风格。

当这几层连接起来，工作台才会真正成为一个能长期运营的内容增长系统。
