# PRD-V3 开发前说明文档

## 1. 文档定位

本文档用于把 `prd-v3-content-growth-workbench.md` 翻译成开发前可执行的理解框架。

它不是代码方案，也不是最终技术设计。它的作用是：

1. 明确哪些需求是 V3 必须实现的业务边界。
2. 明确当前页面职责要如何迁移。
3. 明确哪些旧逻辑需要收敛，避免开发时继续沿用。
4. 明确 Prompt、数据结构、交互和验收的实现口径。

## 2. 开发总原则

V3 开发应遵循四个原则：

1. 先改页面职责，再补复杂能力。
2. 先跑通闭环，再追求智能化。
3. 先让用户看懂系统用什么资料、为什么生成，再接入更自动的语义检索。
4. 不把技术概念暴露给用户，用户只看到业务动作和业务结果。

## 3. 真源文档顺序

开发判断冲突时，以以下顺序为准：

1. `prd-v3-content-growth-workbench.md`
2. `content-growth-system-v2-clarifications.md`
3. `content-growth-system-v2-rules.md`
4. `geo-distilled-terms-refactor.md`
5. 当前代码实现

如果当前代码实现和 PRD-V3 冲突，以 PRD-V3 为准。

## 4. V3 范围解释

### 4.1 必做

| 模块 | 必做内容 |
|---|---|
| 首页 | 改为本月内容增长数据看板 |
| 知识库 | 标准化导入、内容预览、规则切片、Chunk 预览、自动抓取配置 |
| 月度计划 | 只做计划预览，不生成正文 |
| 今日发布 | 选择任务、批量生成正文、确认发布、回填 URL |
| 草稿预览 | 预览、人工修改、AI 二次质检、复制全文 |
| 发布队列 | 只做渠道数据回传 |
| 博客监控 | 诊断页优先、表格靠后、geo optimizer 审计指标 |
| 月度复盘 | 指标图表、蒸馏词矩阵复盘、跳转月度计划预览 |
| Prompt | 五类模板固定进对应流程 |

### 4.2 可后置

| 能力 | 建议阶段 | 原因 |
|---|---|---|
| 向量化 | V3 P2 或 V4 | 基础闭环不应依赖向量库 |
| 语义检索 | V3 P2 或 V4 | 先有可预览 Chunk，再接自动检索 |
| 自动 Chunk 推荐 | V4 | 需要稳定的 Chunk 数据和质量反馈 |
| 真实远程日志 | 条件具备后 | 当前用户拿不到远程服务器真实日志 |
| 全自动发布 | 暂不做 | 当前发布仍应人工控制 |

### 4.3 明确不做

1. 不展示用户可见可信等级。
2. 不让用户在内容生成页选择知识库类型。
3. 不在单篇草稿页做正文生成。
4. 不在月度计划页生成正文或草稿结构。
5. 不在发布队列做发布确认和 URL 回填。
6. 不伪造 CDN、Nginx、AI Bot 真实日志。
7. 不把模型不可观察的内部引用路径写成确定结论。

## 5. 推荐实施阶段

### 5.1 P0：页面职责收敛

目标：先把用户路径从混乱变清楚。

任务：

1. 首页改为数据看板。
2. 月度计划只保留计划预览逻辑。
3. 今日发布承担正文批量生成、确认发布、URL 回填。
4. 草稿预览移除单篇生成心智，聚焦修改、质检、复制。
5. 发布队列收敛为数据回传。

验收：

1. 用户能从月度计划一路走到今日发布、草稿预览、复制发布、回填 URL。
2. 发布队列不再是发布确认入口。
3. 月度计划不再展示正文生成或草稿结构。

### 5.2 P1：诊断页改造

目标：让 GEO 和博客监控先回答“问题在哪里、下一步做什么”。

任务：

2. 增加 Prompt 组抽屉。
3. 蒸馏词默认全选。
4. 增加四层引用结果。
5. 增加测试频率建议。
6. 博客监控改成诊断优先，表格靠后。
7. 加入 geo optimizer 指标。

验收：

2. 博客监控页面先展示问题分布和优先处理问题。
3. 用户能把诊断问题转入候选池或月度计划信号。

### 5.3 P2：知识库与蒸馏词能力增强

目标：提高生成稳定性和长期复盘能力。

任务：

1. 知识库统一导入。
2. 内容预览、规则切片、Chunk 预览。
3. 自动抓取站点并更新知识库。
4. 多模型验证作为蒸馏词生成前置自动动作。
5. 蒸馏词矩阵复盘进入月度复盘和蒸馏词库。
6. 如开发成本允许，再接入语义检索和向量化。

验收：

1. 用户能看到资料被如何解析和切片。
2. 内容生成可以追溯使用的知识片段。
3. 月度复盘能按蒸馏词矩阵解释本月覆盖和缺口。

## 6. 现有页面迁移说明

### 6.1 首页

当前：任务队列和指标卡已经有基础。

迁移：

1. 保留核心入口，但重排为经营看板。
2. 增加本月度计划、已生成、已发布、待回填、待回传。
3. 增加工作台流程结构图。
4. 优先待办按风险排序。

### 6.2 月度计划页

当前：会确认任务进入今日生成，并带有覆盖草稿和发布队列的旧提示。

迁移：

1. 删除或改写“覆盖草稿和发布队列”的危险心智。
2. 月度计划生成后进入预览，不直接影响正文。
3. 确认后只写入本月度计划池。
4. 表格字段精简为标题、渠道、产品、类型、主蒸馏词、来源问题。

### 6.3 今日发布页

当前：支持批量生成和单篇生成，并提示进入终稿确认和发布队列。

迁移：

1. 单篇生成入口下线或降级为开发调试能力，不在用户流程出现。
2. 增加从已确认月度计划选择今日任务的区域。
3. 批量生成成为唯一正文生成动作。
4. 今日列表增加 URL 回填状态。
5. 今日页确认发布后弹窗提醒 URL 回填。

### 6.4 草稿预览页

当前：更像终稿确认页，支持保存、重新生成、加入发布队列。

迁移：

1. 页面名称和文案改成“草稿预览”。
2. 保留保存修改。
3. 移除用户侧重新生成正文入口。
4. 增加 AI 二次质检。
5. 增加人工修改标记和红色未通过片段。
6. 通过后显示复制全文。
7. 未通过时，显示“返回修改前”和“删除红色失败片段”。

### 6.5 发布队列页

当前：仍可标记发布、回填 URL、录入指标。

迁移：

1. 发布确认和 URL 回填迁移到今日发布页。
2. 发布队列页面保留已发布记录和渠道数据导入。
3. 页面文案改为“数据回传”或强化该定位。
4. 匹配失败项给出原因和修复建议。


当前：已有平台、Prompt 组、结果表和候选池动作。

迁移：

1. 把结果表下移。
2. 顶部增加诊断摘要和四层引用指标。
3. Prompt 组做抽屉管理。
4. 蒸馏词默认全选。
5. 自动化配置显示建议频率、上次运行、下次运行、手动运行。

### 6.7 博客监控页

当前：导入区域和博客列表权重较高。

迁移：

1. 保留导入能力，但不作为第一视觉。
2. 增加诊断摘要、问题分布和优先处理问题。
3. 表格靠后。
4. 标题解析失败时标题留空，URL 放详情。
5. 不显示“待解析标题”，不增加多余重新解析动作。

### 6.8 月度复盘页

当前：已有月度复盘生成和下月计划草稿动作。

迁移：

1. 直接“生成下月计划草稿”改为“进入月度计划生成预览”。
2. 增加图表和指标总览。
3. 增加蒸馏词矩阵复盘。
4. 月度复盘只生成信号，不直接覆盖计划。

## 8. Prompt 固定方式

V3 不应把 Prompt 写成散落在各处的字符串。建议按模板注册。

### 8.1 五类模板对应关系

| 模板 | 使用页面 | 调用时机 |
|---|---|---|
| 月度计划生成模板 | 月度计划 | 一键生成计划预览 |
| 渠道标题模板 | 月度计划 | 生成或编辑标题 |
| 证据选择模板 | 今日发布 | 批量生成前 |
| 批量正文生成模板 | 今日发布 | 批量生成正文 |
| AI 二次质检模板 | 草稿预览 | 人工修改后 |

### 8.2 稳定性要求

每个模板必须满足：

1. 输入结构固定。
2. 输出结构固定。
3. 失败原因可读。
4. 不满足硬约束时返回错误，不硬编。
5. 需要保留 Prompt 版本号，方便后续复盘。



规则：

| 每月发布量 | 建议频率 |
|---|---|

额外触发条件：

1. 官网上新一批博客。
2. 官网首页或产品页改版。
3. 新蒸馏词矩阵上线。
4. 重要活动周或发布周。

页面需要显示：

1. 当前建议频率。
2. 上次测试时间。
3. 下次建议测试时间。
4. 手动运行按钮。
5. 自动化启用 / 停用。

## 10. 博客监控 geo optimizer 接入说明

业务目标不是为了多一个检测指标，而是为了判断官网博客是否适合作为 AI 信源。

建议检测维度：

1. 页面可访问性。
2. AI crawler 可访问性。
3. 页面标题和正文可提取性。
4. 结构化内容完整度。
5. FAQ / How-to / Schema 完整度。
6. 明确结论是否足够。
7. 官网事实和产品指向是否足够。
8. Chunk 准备度。

页面呈现：

1. 先展示诊断摘要。
2. 再展示问题分布。
3. 再展示优先处理问题。
4. 最后展示博客列表。

## 11. 设计实现口径

### 11.1 页面设计原则

1. B 端工作台，不做营销页。
2. 蓝色为主色。
3. 指标优先于表格。
4. 诊断优先于筛选。
5. 主动作只有一个。
6. 圆角统一 8px 以内。
7. 卡片少而准，不做卡套卡。
8. 表格要紧凑，但不要压迫。

### 11.2 页面通用结构

```text
PageHeader
-> Metric Strip
-> Diagnosis Summary
-> Recommended Actions
-> Toolbar / Filters
-> Detail Table
```

### 11.3 状态颜色

| 状态 | 颜色方向 |
|---|---|
| 通过 / 健康 / 已闭环 | 绿色 |
| 待处理 / 引用不足 / 待回填 | 橙色 |
| 阻断 / 失败 / 高风险 | 红色 |
| 信息 / 运行中 / 默认操作 | 蓝色 |
| 次级分类 | 青色或紫色少量使用 |

## 12. 验收测试建议

### 12.1 主链路测试

必须验证：

1. 生成月度计划预览。
2. 修改并确认月度计划。
3. 今日页选择任务。
4. 批量生成正文。
5. 进入单篇草稿预览。
6. 人工修改。
7. AI 二次质检。
8. 未通过时不能复制全文。
9. 删除红色失败片段后重新质检。
10. 通过后复制全文。
11. 今日页确认已发布。
12. 回填 URL。
13. 发布队列导入渠道数据。
14. 月度复盘读取数据。

### 12.2 页面职责测试

必须确认：

1. 月度计划页没有正文生成入口。
2. 单篇草稿页没有单篇生成入口。
3. 发布队列没有 URL 回填主入口。
4. 博客监控先展示诊断。

### 12.3 规则测试

必须确认：

1. 月度计划使用四层约束模型。
2. GEO 引用输出四层结果。
3. Prompt 组和蒸馏词不是一对一绑定。
4. 蒸馏词默认全选。
5. 多模型验证自动生成蒸馏词状态。

## 13. 开发风险和建议

### 13.1 最大风险

最大风险不是技术实现，而是旧页面职责残留。

例如：

1. 月度计划还在管正文。
2. 发布队列还在回填 URL。
3. 草稿页还在单篇生成。
4. GEO 和博客监控还是表格优先。

这些残留会让用户继续混乱。

### 13.2 建议开发顺序

建议先做：

1. 页面职责和文案调整。
2. 主链路状态迁移。
3. 再做知识库 Chunk 和 Prompt 模板。
4. 最后做图表、自动化和语义检索增强。

原因：

页面职责是产品骨架。如果骨架不对，智能能力越多，用户越不清楚系统在做什么。

## 14. 开发前最终确认

V3 的核心验收不是“增加了多少功能”，而是用户能不能用一条稳定路径完成：

```text
资料沉淀
-> 月度计划预览
-> 今日批量生成
-> 单篇修改质检
-> 人工发布回填 URL
-> 渠道数据回传
-> 博客监控和 GEO 诊断
-> 月度复盘
-> 反哺下月计划
```

只要这条链路清楚，后续再接入向量化、语义检索、真实日志和更强自动化，系统才会越做越稳。
