# 分支二：月度策略与批量生产开发文档

## 1. 分支目标

分支：`codex/v5-monthly-strategy-production`

本分支把问题、知识和表达预设转换为可批准、可生成、可排程的月度内容：

1. 月度内容矩阵承担`内容策略包`的生成、编辑和批准。
2. 用户按`目标问题 + 文章类型 + 每渠道配额`编排，不展示内容角色或子意图。
3. 多选渠道时，配额表示每个渠道分别生成的数量。
4. `批量生成中心`作为月度内容矩阵子入口，不再是顶层导航。
5. 批量生成中心只执行已批准策略，系统自动完成生成、检查、修复和失败重试，用户只预览内容或安排发布。
6. 正文详情从任务行打开抽屉或全屏预览，不建立独立导航页面。

本分支依赖分支一已合并后的问题版本、知识快照和文章表达预设契约。本分支不实现 AI 前台采集、月度效果归因或官网审计。

## 2. 核心业务规则

### 2.1 配额语义

```text
渠道成品数 = 每渠道配额 x 所选渠道数
月度渠道成品总数 = 所有配额行的渠道成品数之和
```

示例：

```text
目标问题：企业应该如何选择腾讯云 ADP 服务商？
文章类型：选型与比较
每渠道配额：4
渠道：官网、知乎

结果：官网 4 篇 + 知乎 4 篇 = 8 篇渠道成品
```

若用户希望两个渠道合计 4 篇，应展开渠道设置为官网 2、知乎 2。页面不能把“4 篇”自动平均分配，也不能在多选渠道后隐藏总量增长。

### 2.2 用户可见编排维度

用户只配置目标问题、文章类型、每渠道数量、渠道、产品表达规则包、知识库和文章表达预设。

关键词覆盖、Evidence Preview、去重和证据检查由系统承担。系统内部可以保留 ContentCluster 或 QuestionFacet，但页面不以此要求用户做决策。

## 3. 页面与路由

| 页面 | 正式路由 | 兼容入口 | 本分支职责 |
|---|---|---|---|
| 月度内容矩阵 | `/monthly-matrix` | 保留 | 月份、目标问题、内容策略包、矩阵版本 |
| 内容策略包 | `/monthly-matrix`默认工作区 | 旧策略路径重定向 | 配额编辑、证据预检、批准 |
| 批量生成中心 | `/monthly-matrix/batch-generation` | `/batch-generation`重定向 | 内容生成与发布排程 |

| 当日执行 | `/daily-execution` | 保留 | 暂不改变职责 |
| 数据回传 | `/publish` | 保留 | 暂不改变职责 |

## 5. 月度矩阵任务

策略批准后，一条“ADP 选型 / 选型与比较 / 官网 4 / 知乎 4”配额生成 4 个基础选题建议，每个基础选题展开官网和知乎版本，共 8 个 `ContentMatrixItem`。

同一基础选题的渠道版本共享目标问题和可复用证据候选，但标题、正文、格式、CTA 和发布记录独立。

```text
+--------------------------------------------------------------------------------+
| 矩阵任务 30                          [目标问题 v] [类型 v] [渠道 v] [状态 v]   |
+--------------------------------------------------------------------------------+
| 基础选题 / 渠道版本                    类型       证据状态      状态       操作 |
| 如何评估 ADP 服务商的实施能力                                                 |
|   ├ 官网：企业选择 ADP 服务商应看哪些实施能力  选型比较   [通过]      待批准 [详情]|
|   └ 知乎：选 ADP 服务商时最容易忽略什么       选型比较   [通过]      待批准 [详情]|
```

正文通过系统检查后自动进入待排程，不要求用户逐篇确认。用户可按需查看内容依据或主动编辑；编辑保存后系统在后台重新检查。正文预览不是独立导航页面，可用`?draftId=`恢复打开状态。

后台仍保存生成输入、EvidencePack、检查结果和必要审计记录，但这些技术信息不进入正文页面。点击`编辑正文`直接进入当前正文编辑状态，保存和历史保护由系统自动完成，不要求用户创建或命名版本。

### 6.4 自动检查与恢复规则

自动检查是后台步骤，不设置独立页面：

```text
生成正文
-> 检查事实依据、公开范围、禁止表达、结构完整和渠道适配
-> 可自动修复：自动改写并再次检查，最多 2 轮
-> 技术失败：指数退避自动重试，最多 3 次
-> 检查通过：状态改为可用，自动进入待排程
-> 缺少关键事实依据：状态改为待补资料，关联知识库待处理事项
-> 多次技术失败：批次结束后统一自动重试并记录服务状态
```

用户不需要看到硬规则清单、软质量分数、模型错误、EvidencePack 或 Claim 等内部名词。结构、自然度、重复度和渠道适配只作为系统自动优化信号，不要求用户决策。

系统处理边界：

1. 可以通过删除无证据表述、增加条件或改写结构解决的问题，系统自动修复。
2. 明确为内部资料的内容自动排除，不要求用户再次确认公开范围。
3. 缺少非关键细节时使用已有证据生成，不阻断整篇正文。
4. 缺少文章主题成立所必需的关键事实时才显示`待补资料`。
5. 该状态只提供一个业务动作：打开知识库对应待处理事项；资料更新后系统自动重检，不显示手动重试按钮。
6. 技术失败由系统恢复并记录，只有持续影响整个批次时才在页面顶部显示服务状态，不建立逐条异常队列。

## 7. 状态与约束

策略包状态：

```text
draft -> preview_checking -> pending_review
-> approved / partially_approved -> superseded
```

矩阵任务状态：

```text
draft -> evidence_preview_ready / evidence_gap -> approved
-> ready_for_generation -> generating -> available / awaiting_material
-> scheduled
```

强制约束：

1. 配额展开总数不得大于月度渠道成品总数。
2. 提交审核时必须等于月度总数；草稿可以小于总数。
3. 规则包、知识索引和 EvidencePack 必须绑定同一 `sourceSnapshotHash`。
4. 只有批准矩阵项可以进入批量生成。
5. 只有`generatable`任务调用正文模型；可自动降级的表述由系统先处理，不要求人工确认。
6. 生成时冻结表达预设版本，不读取后来修改的默认值。
7. 渠道版本不可直接复制同一正文，必须经过渠道表达编译和后台自动检查。

## 10. 开发顺序

1. 同步分支一合并后的最新`main`。
2. 扩展 MonthlyPlan 和 ContentQuotaRule 契约。
3. 重构月度计划配置与策略包编辑。
4. 实现渠道配额展开、总量校验和证据预检。
5. 实现策略批准与矩阵任务展开。
6. 将批量生成中心迁入月度内容矩阵子路由。
7. 实现按问题和类型分组的内容与排程，以及后台自动修复和重试。
8. 实现精简正文预览抽屉、按需编辑和可恢复深链接。
9. 更新结构、交互、工作流和浏览器测试。

## 11. 验收标准

1. 页面不要求用户配置内容角色或子意图。
2. “配额 4 + 官网、知乎”明确展开为 8 篇渠道成品。
3. 用户可关闭相同配额并分别设置官网 2、知乎 2。
4. 草稿实时显示月度总数、已分配和待分配；审核前合计必须相等。
