# 分支三：AI 前台测试、月度复盘与官网审计开发文档

## 1. 分支目标

本分支承接已发布内容之后的观察、归因和复盘，不负责重新定义知识治理或月度生产规则。核心交付是：

1. 将原“AI 前台可见度”统一命名为“AI 前台测试”。
2. P0 建立可审计的单次采集任务，保存 AI 回答、引用证据、截图和采集条件。
3. 支持用户从历史任务中任选两次进行问题级对比，不预设固定日期变化计划。
4. 将回答表现拆解为内容缺口和证据缺口，分别进入博客候选池或知识库问题与冲突。
5. 将月度复盘放在博客候选池之后，以问题为主线关联内容、发布、指标和 AI 回答。
6. P1 将品牌官网审计合并进官网博客监控的“官网审计”页签，不建立独立导航。

本分支禁止：

- 自动把观察结果升级为事实、Claim 或月度生产任务。
- 把浏览器 Cookie、密码、Token 或完整会话上传到服务端。
- 将 Computer Use 作为常规批量采集主链路。
- 在 P0 创建 D3、D7、D14、D30 等固定日期自动采集计划。
- 建立独立的官网审计导航、独立规划周期或独立复盘周期。

## 2. 用户可见结果

用户完成一次闭环时，只需要做五个判断：

```text
选择问题和 AI 平台
-> 启动单次采集
-> 查看回答与引用证据
-> 任选另一次历史任务进行对比
-> 确认缺口去向
```

系统负责：

- 检查浏览器伴侣与本地 Runner 是否就绪。
- 在用户已登录的 AI 页面执行问题并监听流式回答。
- 捕获回答 DOM、可见文本、引用 URL、标题、截图和采集上下文。
- 对原始采集包计算 SHA-256，保留不可变原始记录。
- 将回答陈述与当前 Snapshot 下的 Claim、Chunk、Source 进行匹配。
- 产生候选缺口和解释，不替用户做最终业务判断。
- 聚合问题在整个月份中的发布、指标、AI 回答和候选动作。

### 5.3 任务状态与失败反馈

```text
draft
-> environment_checking
-> queued
-> waiting_for_browser
-> submitting_prompt
-> streaming
-> stabilizing
-> capturing
-> completed
```

任一步骤可进入：

```text
needs_login / adapter_mismatch / interrupted / timed_out / capture_failed
```

失败信息必须包括：

- 失败阶段和可读原因。
- 已保留的数据范围。
- 是否可从当前阶段继续。
- 用户下一步动作，例如“在浏览器中重新登录后重试”。

平台页面结构变化导致适配器失效时，任务应停止并标记 `adapter_mismatch`，不得用不可靠的选择器拼接出一个“成功”回答。

### 5.6 缺口分析与定义

系统只生成候选缺口，人工复核后才能进入下游：

| 缺口代码 | 判断问题 | 默认去向 |
| --- | --- | --- |
| `answer_coverage_gap` | AI 回答缺少用户希望覆盖、且已有证据支持的重要内容吗？ | 博客候选池 |
| `citation_gap` | 回答提到了主体或能力，但没有引用对应官方或自有内容吗？ | 博客候选池 / 官网监控 |
| `evidence_gap` | 希望表达的能力缺少可公开 Source 或已验证 Claim 吗？ | 知识库问题与冲突 |
| `relationship_gap` | 主体间合作、实施或归属关系是否缺少清晰证据？ | 知识库问题与冲突 |
| `freshness_gap` | 引用或知识证据是否过期、版本不一致？ | 知识库问题与冲突 |
| `entity_gap` | 目标实体是否未进入回答或被错误归属？ | 博客候选池 |
| `observation_uncertain` | 页面结构、回答解析或证据映射是否不够可靠？ | 人工复核队列 |

同一个回答可以有多个缺口。`citation_gap`不等于`evidence_gap`：前者可能已有合格证据，但 AI 没有引用；后者表示平台自身尚不能支撑希望发布的事实。

示例：

```text
问题：企业应该如何选择腾讯云 ADP 服务商？

采集结果：
- AI 提到实施经验、知识治理、安全和持续运营。
- 回答未出现 JOTO。
- 引用了腾讯云和第三方内容，没有引用 JOTO 官网。

平台证据：
- 已验证 Claim：JOTO 可在约定项目范围内提供 ADP 实施与交付。
- 已有实施流程 Chunk。
- 没有公开案例或量化效果 Claim。

候选结论：
- entity_gap：JOTO 未进入候选服务商表达。
- citation_gap：没有引用 JOTO 的可公开页面。
- answer_coverage_gap：实施交付表达没有覆盖 JOTO 已有证据。
- evidence_gap：公开案例与效果数据不足，不能生成相关承诺。
```

## 6. 采集技术实现

### 6.1 推荐架构

P0 采用以下组合：

```text
V5 Web 应用
  | 创建任务 / 查看结果
  v
采集任务服务与审计存储
  ^ WebSocket 任务状态与结果
  |
本地采集 Runner（127.0.0.1）
  ^ 本机 WebSocket
  |
Chrome Manifest V3 浏览器伴侣
  |
平台适配器 + content script
  v
用户真实登录的 AI 平台页面
```

底层原因：主流 AI 前台依赖登录状态、流式 DOM、动态引用浮层和反自动化策略。浏览器伴侣可以复用用户主动登录的真实页面上下文；本地 Runner 负责排队、状态机、超时、校验和上传经过筛选的采集结果。相比纯服务端爬虫，登录和页面交互更稳定；相比把 Computer Use 作为主链路，结构化 DOM 采集更可验证、成本更低。

### 6.2 浏览器伴侣职责

浏览器伴侣使用 Chrome Manifest V3：

- `host_permissions`只配置明确支持的 AI 平台域名。
- content script 通过平台适配器定位输入框、提交按钮、回答容器、停止按钮和引用节点。
- service worker 管理任务握手、标签页权限和与本地 Runner 的连接。
- 每次任务使用独立 `captureSessionId`，拒绝来自非目标标签页的数据。
- 不读取或传输 Cookie、密码、Local Storage Token。
- 截图只在任务页面执行，并在上传前关联任务 ID、页面尺寸和时间。

平台适配器接口建议：

```ts
interface FrontendCaptureAdapter {
  platform: AiFrontendPlatform;
  version: string;
  detectPage(): AdapterHealth;
  createConversation(): Promise<void>;
  submitQuestion(question: string): Promise<void>;
  observeAnswer(onEvent: (event: StreamEvent) => void): StopObserver;
  extractAnswer(): CapturedAnswer;
  extractCitations(): CapturedCitation[];
  sanitizeArtifact(input: RawArtifact): SanitizedArtifact;
}
```

### 6.3 流式完成判断

content script 使用`MutationObserver`监听回答容器，组合判断完成：

```text
检测到回答节点
-> 持续接收 DOM 变化
-> 平台停止生成按钮消失或完成标记出现
-> 可见文本在稳定窗口内不再变化
-> 引用节点完成解析
-> 进入捕获
```

建议稳定窗口为 1.5 至 3 秒，由适配器配置；同时设置首字超时和总任务超时。只满足“文本不再变化”但停止按钮仍存在时不能判定成功，避免把网络卡顿误认为回答结束。

### 6.4 原始采集包

每次平台任务生成独立不可变原始包：

```text
taskId
captureSessionId
questionVersionId / temporaryQuestionText
platform / modelLabel
locale / region / conversationMode / personalizationMode
adapterVersion / browserVersion
startedAt / completedAt
answerHtmlSanitized / answerText
citations[]: label, url, title, visibleSnippet, position
screenshotArtifactId
completionSignals
captureWarnings[]
```

服务端对规范化后的 manifest 与 artifact 哈希列表计算 SHA-256。原始包不被人工编辑；解析版本、陈述映射、缺口分析和人工复核分别建立版本记录。

### 6.5 Computer Use 的边界

Computer Use 只用于：

- 平台页面发生变化后，工程人员辅助定位新交互。
- 无法自动恢复的登录、验证码或同意弹窗由用户接管。
- P1 之前验证尚未正式支持的平台。

它不用于常规无人值守批量采集，也不绕过验证码、访问控制或平台条款。人工接管后必须重新运行环境检查，并在任务中记录 `manual_intervention`。

## 10. 权限、隐私与失败回退

1. 创建采集任务、查看回答、复核缺口、查看原始工件分别授权。
2. 浏览器伴侣只在用户明确发起的目标任务中读取指定页面。
3. 上传前清除输入框历史、账号菜单、侧栏会话标题等与任务无关的 DOM。
4. 截图默认对账号名称、头像、会话历史和通知信息做遮罩。
5. 不记录 Cookie、密码、Token、浏览器自动填充信息或私有请求头。
6. Runner 离线时任务停留在`waiting_for_browser`，平台服务端不得尝试模拟登录。
7. 引用 URL 不可访问时仍保存原始链接和捕获位置，但标记为`unverified`。
8. 证据映射失败时标记`observation_uncertain`，不得编造 Claim 关联。
9. 原始工件保留周期和删除策略由组织配置；删除必须留审计记录。

## 13. 验收标准

1. 所有用户可见名称统一为“AI 前台测试”，旧名称只存在于迁移说明。
2. P0 只能新建立刻执行的单次采集任务，不出现固定日期自动计划。
3. 用户可从同一问题的历史记录中任选两次任务进行对比。
4. 条件不一致的任务仍可对比，但页面明确显示差异且不生成趋势结论。
5. 浏览器伴侣不上传 Cookie、密码、Token 或非任务页面内容。
6. 流式回答必须通过停止信号、文本稳定窗口和超时组合判断完成。
7. 原始回答、引用、截图、采集条件、适配器版本和 SHA-256 可追溯。
8. 原始回答不可编辑；系统分析和人工复核独立版本保存。
9. 七类缺口均有定义、证据位置、置信度和人工去向。
10. 内容缺口进入博客候选池，证据缺口进入知识库问题与冲突；均不自动创建月度任务。
11. 月度复盘按目标问题关联计划、成品、发布、指标和 AI 采集结果。
12. 下月建议只生成 Proposal，必须经下个月 MonthlyPlan 人工批准。
13. P1 官网审计位于`/blog-monitor?tab=site-audit`，侧栏没有独立官网审计入口。
14. 官网审计和 AI 前台测试不合并状态、指标或总分。
15. 桌面端线框中的文字、状态和操作入口无重叠或不可达操作。
16. `npm.cmd run typecheck`、`npm.cmd run validate:structure`、相关 smoke 和生产构建通过。
17. 维护代码、文档、脚本和状态夹具中不存在独立周规划或周复盘标识。

## 14. 分支交付说明

本分支应在分支一和分支二合并后再进行正式集成。它读取已发布内容、正式问题版本、知识 Snapshot 和月度计划结果，但不回写已批准策略包或重新计算渠道配额。

P0 验收应至少打通一个正式支持平台的真实前台垂直链路。其他平台在适配器未达到可靠性标准前显示“尚未支持”，不能用静态模拟结果替代真实采集。
