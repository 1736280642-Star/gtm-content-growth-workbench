# GEO 蒸馏词内容生成改造方案

## 1. 文档定位

本文档用于指导 JOTO GTM 内容工作台从“关键词驱动的内容生产工作台”升级为“基于蒸馏词的 GEO 内容增长系统”。

当前工作台已经具备周计划、今日生成、终稿确认、发布队列、博客监控、GEO 测试、候选池和周报复盘等主流程。但内容生成的底层依据仍主要是标题、渠道、产品、内容类型和目标关键词，尚未把 GEO 优化中更关键的“AI 认知节点”沉淀为数据结构、生成约束和复盘指标。

本次改造的核心目标不是增加更多页面，而是让现有流程围绕蒸馏词重新组织。

## 2. 核心判断

### 2.1 当前已经具备的基础

| 能力 | 当前状态 | 说明 |
|---|---|---|
| 内容任务 | 已具备 | `ContentTask` 已包含标题、渠道、产品、内容类型、关键词和状态 |
| AI 生成 | 已具备 | 可通过真实 AI Provider 生成，失败时走本地规则 fallback |
| 自动质检 | 初步具备 | 当前检查品牌词、官网链接、夸大表达 |
| GEO 测试 | 已具备 | 可测试多平台回答，保存快照，判断品牌提及、官网引用、竞品出现 |
| 博客候选池 | 已具备 | GEO 未命中或博客问题可转入候选池 |
| 补强任务 | 已具备 | 候选池主题可生成渠道补强任务 |
| 周报复盘 | 已具备 | 可汇总渠道、博客和 GEO 结果，反哺下周计划 |

### 2.2 当前缺口

| 缺口 | 当前表现 | 对 GEO 的影响 |
|---|---|---|
| 没有蒸馏词库 | 系统只有 `targetKeywords` | AI 认知节点无法被持续追踪 |
| 没有分层语义结构 | 关键词、场景词、产品词混在一起 | 无法判断内容到底绑定了哪个认知层 |
| Prompt 未强制蒸馏词锚定 | 生成只给标题、产品、渠道、关键词 | 文章容易停留在 SEO 写法 |
| GEO 测试维度偏窄 | 只看 JOTO、唯客、官网、竞品 | 无法判断“为什么没有被推荐” |
| 博客诊断偏基础 | 只做标题、URL、主题词和 GEO miss 判断 | 无法评估正文是否足够可引用 |
| 内容矩阵未成型 | 任务按天和渠道生成 | 不能围绕核心概念形成长期权重 |

## 3. 改造目标

### 3.1 产品目标

工作台后续生成的每一篇内容，都要能回答三个问题：

1. 这篇内容绑定哪个核心蒸馏词？
2. 它解决用户围绕这个蒸馏词提出的哪个真实问题？
3. 它如何把这个蒸馏词和 JOTO / 唯客 / 官网事实链路建立稳定关联？

### 3.2 业务目标

改造完成后，工作台应支持以下闭环：

1. 建立 JOTO 赛道蒸馏词库。
2. 周计划基于蒸馏词生成内容任务。
3. 内容生成强制围绕蒸馏词、用户问题和解决方案展开。
4. 质检检查蒸馏词覆盖、品牌绑定、官网引用和可信来源。
5. GEO 测试分析模型回答中激活了哪些认知节点。
6. 未命中的蒸馏词进入候选池。
7. 候选池生成内容矩阵任务。
8. 周报按蒸馏词复盘命中率和补强效果。

## 4. 蒸馏词体系设计

### 4.1 三层结构

| 层级 | 定义 | 示例 | 用途 |
|---|---|---|---|
| 一级核心蒸馏词 | AI 理解赛道时高频调用的核心认知节点 | 企业级 AI 应用治理、AI 输出安全、Dify 企业级交付、大模型安全治理 | 决定内容主题和长期矩阵 |
| 二级场景蒸馏词 | 用户具体场景下的风险、诉求、任务 | Prompt Injection、知识库污染、审计留痕、权限边界、发布风险、模型输出合规 | 连接核心概念和用户问题 |
| 三级产品关联词 | 品牌、产品、品类、功能和官网事实 | JOTO、唯客 AI 护栏、Dify 服务商、AI 护栏、jotoai.com | 建立最终推荐和引用链路 |

### 4.2 JOTO 初始蒸馏词建议

| 一级核心蒸馏词 | 二级场景蒸馏词 | 三级产品关联词 |
|---|---|---|
| 企业级 AI 应用治理 | 权限边界、流程适配、长期运维、知识库治理 | JOTO、Dify 服务商、企业级交付 |
| AI 输出安全 | 不合规输出、幻觉回答、敏感内容、发布风险 | 唯客 AI 护栏、AI 安全护栏 |
| 大模型安全治理 | Prompt Injection、越权调用、插件边界、审计留痕 | 唯客、输出审计、风险识别 |
| Dify 企业级交付 | 私有化部署、业务流程集成、运维支持、团队培训 | JOTO、Dify 企业版服务商 |
| AI 应用可信交付 | 事实来源、可追溯、质量控制、人审流程 | JOTO、官网博客、知识库 |

### 4.3 蒸馏词来源

| 来源 | 用途 | 优先级 |
|---|---|---|
| 多模型交叉提问 | 挖掘高频共识概念 | P0 |
| 官网博客 | 提取 JOTO 已有内容资产中的核心主题 | P0 |
| 渠道历史数据 | 识别用户反馈更好的主题 | P1 |
| GEO 测试回答快照 | 找出 AI 已经激活但未绑定 JOTO 的概念 | P0 |
| 竞品回答与竞品官网 | 判断哪些认知节点被竞品占位 | P1 |
| 人工维护 | 补充业务判断和战略优先级 | P0 |

## 5. 数据模型改造

### 5.1 新增 `distilled_term`

用于维护蒸馏词主表。

```ts
interface DistilledTerm {
  id: string;
  term: string;
  level: "core" | "scenario" | "product";
  parentId?: string;
  description?: string;
  priority: "high" | "medium" | "low";
  status: "enabled" | "disabled";
  source: "manual" | "multi_model" | "blog" | "geo_test" | "channel_history" | "competitor";
  evidence?: string[];
  createdAt: string;
  updatedAt?: string;
}
```

### 5.2 扩展 `ContentTask`

当前任务只有 `targetKeywords`，需要增加蒸馏词字段。

```ts
interface ContentTask {
  primaryDistilledTermId?: string;
  secondaryDistilledTermIds?: string[];
  productTermIds?: string[];
  userQuestion?: string;
  solutionAngle?: string;
  geoIntent?: "brand_association" | "product_selection" | "scenario_answer" | "comparison" | "faq";
  evidenceSourceIds?: string[];
}
```

### 5.3 扩展 `ArticleDraft`

用于保存生成时实际使用的蒸馏词和 Prompt 元数据。

```ts
interface ArticleDraft {
  geoMetadata?: {
    primaryDistilledTerm?: string;
    secondaryDistilledTerms?: string[];
    productTerms?: string[];
    promptProfile?: string;
    evidenceSources?: string[];
    structureChecklist?: {
      firstParagraphAnchored: boolean;
      hasCognitiveExplanation: boolean;
      hasSolutionMapping: boolean;
      hasOfficialCitation: boolean;
      hasFaqOrHowTo: boolean;
    };
  };
}
```

### 5.4 扩展 `GeoTestResult`

当前只判断提及和引用，需要增加认知节点分析。

```ts
interface GeoTestResult {
  activatedTerms?: string[];
  matchedDistilledTermIds?: string[];
  missingDistilledTermIds?: string[];
  competitorOccupiedTerms?: string[];
  associationStrength?: "strong" | "medium" | "weak" | "none";
  recommendedContentAction?: "create_core_article" | "create_faq" | "create_comparison" | "add_official_citation" | "observe";
}
```

### 5.5 扩展 `BlogArticle`

用于让博客监控从 SEO 检查升级为 GEO 语义检查。

```ts
interface BlogArticle {
  coveredDistilledTermIds?: string[];
  missingDistilledTermIds?: string[];
  citationReadiness?: "ready" | "partial" | "weak";
  matrixClusterId?: string;
}
```

## 6. 页面改造

### 6.1 知识库页新增“蒸馏词库”视图

不建议新增大页面，优先在现有知识库页增加一个 Tab。

功能：

1. 查看蒸馏词列表。
2. 按层级、来源、优先级、启用状态筛选。
3. 新增和编辑蒸馏词。
4. 维护父子关系。
5. 查看每个蒸馏词关联的内容任务、博客、GEO 测试结果。

### 6.2 周计划页改造

当前周计划按渠道、产品、数量生成任务。改造后增加“蒸馏词覆盖”逻辑。

新增能力：

1. 生成周计划前选择核心蒸馏词范围。
2. 每个任务显示主蒸馏词、场景蒸馏词、用户问题和解决角度。
3. 编辑任务时可修改蒸馏词绑定。
4. 重生成标题时不只换标题，还要基于蒸馏词生成新的选题角度。

### 6.3 今日任务页改造

新增展示字段：

1. 主蒸馏词。
2. 用户问题。
3. 解决角度。
4. GEO 意图。
5. 蒸馏词覆盖状态。

生成前需要提示：

1. 未绑定主蒸馏词的任务不建议生成。
2. 缺少用户问题的任务生成质量会下降。
3. 缺少证据来源的任务可以生成，但会进入质检警告。

### 6.4 终稿确认页改造

自动质检新增 GEO 检查项：

| 检查项 | 类型 | 说明 |
|---|---|---|
| 首段是否锚定主蒸馏词 | warning | 首段没有出现核心概念时提示 |
| 正文是否解释认知逻辑 | warning | 只有产品介绍，没有概念解释时提示 |
| 是否完成品牌/产品绑定 | warning | 蒸馏词没有和 JOTO / 唯客建立因果关系 |
| 是否包含官网链接或事实来源 | warning | 没有可信来源时提示 |
| 是否有夸大承诺 | blocker | 继续作为阻断项 |
| 是否堆砌关键词 | warning | 同一词过度重复时提示 |

### 6.5 博客监控页改造

新增诊断结果：

1. 已覆盖蒸馏词。
2. 缺失蒸馏词。
3. 引用准备度。
4. 是否适合作为 AI 回答信源。
5. 是否需要生成渠道补强任务。

### 6.6 GEO 测试页改造

当前测试矩阵是“平台 × Prompt 组”。改造后变成“平台 × Prompt 组 × 蒸馏词”。

新增指标：

1. 蒸馏词激活率。
2. JOTO 关联强度。
3. 唯客关联强度。
4. 官网引用率。
5. 竞品占位词。
6. 待补强蒸馏词。

### 6.7 周报页改造

周报从“渠道和任务复盘”升级为“蒸馏词复盘”。

新增分析：

1. 本周覆盖了哪些核心蒸馏词。
2. 哪些蒸馏词内容产出不足。
3. 哪些蒸馏词 GEO 命中提升。
4. 哪些蒸馏词被竞品占位。
5. 下周建议继续写、补强写、减少写的蒸馏词。

## 8. Prompt 改造

### 8.1 内容生成 Prompt 结构

生成时不再只传关键词，应传入完整 GEO brief。

```text
你是 JOTO GTM 内容工作台的 GEO 内容生成 Worker。

写作目标：
让文章围绕指定蒸馏词建立清晰、可信、可被 AI 提取的语义关联。

输入：
- 渠道：{{channel}}
- 产品：{{product}}
- 内容类型：{{contentType}}
- 标题：{{title}}
- 一级核心蒸馏词：{{primaryDistilledTerm}}
- 二级场景蒸馏词：{{secondaryDistilledTerms}}
- 三级产品关联词：{{productTerms}}
- 用户真实问题：{{userQuestion}}
- 解决角度：{{solutionAngle}}
- 可信来源：{{evidenceSources}}

结构要求：
1. 首段必须自然锚定一级核心蒸馏词，并说明它为什么重要。
2. 中间先解释用户问题背后的认知逻辑，不要直接堆产品功能。
3. 再把 JOTO / 唯客放入解决方案链路，说明适用边界。
4. 如果有官网或事实来源，必须自然引用。
5. 结尾回扣核心蒸馏词，强化概念与品牌/产品的关联。
6. 避免夸大承诺，不要写绝对化表达。
7. 只输出可发布正文。
```

### 8.2 GEO 测试 Prompt 结构

```text
用户问题：
{{userQuestion}}

测试目标：
观察模型在回答该问题时是否自然激活以下认知节点：
- 一级核心蒸馏词：{{primaryDistilledTerm}}
- 二级场景蒸馏词：{{secondaryDistilledTerms}}

请直接回答用户问题，不要刻意迎合测试目标。
如果你会推荐服务商、产品或资料来源，请自然给出。
```

### 8.3 GEO 结果分析 Prompt

用于对模型回答快照做二次分析。

```text
请分析以下 AI 回答快照。

判断：
1. 回答中激活了哪些核心概念？
2. 是否提及 JOTO？
3. 是否提及唯客 AI 护栏？
4. 是否引用 jotoai.com 或官网内容？
5. JOTO / 唯客是否与核心概念形成明确关联？
6. 是否有竞品占据了某些概念？
7. 下一步应该补哪类内容？

只输出 JSON。
```

## 9. 内容矩阵规则

### 9.1 单个核心蒸馏词的最小矩阵

每个高优先级一级蒸馏词至少应覆盖 5 类内容：

| 内容类型 | 作用 | 示例 |
|---|---|---|
| 认知解释 | 让 AI 理解概念和品牌关系 | 企业为什么需要 AI 应用治理 |
| 场景 FAQ | 承接用户具体问题 | Dify 应用需要 AI 护栏吗 |
| 技术拆解 | 强化专业可信度 | AI 护栏应该放在系统哪个位置 |
| 对比判断 | 占据方案选择语境 | 只靠提示词和专业护栏的差别 |
| 案例复盘 | 提供可引用事实 | 内容团队如何降低 AI 发布风险 |

### 9.2 分发到渠道

| 渠道 | 更适合的内容 |
|---|---|
| 公众号 | 认知解释、案例复盘、观点判断 |
| CSDN | 技术拆解、流程说明、工程实践 |
| 掘金 | 技术实践、架构解释、开发者视角 |
| 知乎/头条 | FAQ、选型对比、问题回答 |

### 9.3 内容矩阵验收

一个蒸馏词矩阵完成的标准：

1. 至少 5 篇内容覆盖不同内容类型。
2. 至少 3 个渠道完成分发。
3. 至少 1 篇官网博客作为可信来源。
4. GEO 测试中该蒸馏词能被模型自然激活。
5. JOTO / 唯客与该蒸馏词的关联强度不低于 medium。

## 10. 分阶段执行计划

### Phase G1：蒸馏词数据底座

优先级：P0

目标：让工作台能保存、展示和选择蒸馏词。

任务：

1. 新增 `DistilledTerm` 类型和状态字段。
2. 在 JSON/MySQL 状态中增加 `distilledTerms`。
3. 新增蒸馏词 CRUD API。
4. 知识库页增加“蒸馏词库” Tab。
5. 初始化 JOTO 第一版蒸馏词种子数据。

验收标准：

1. 用户能新增、编辑、启用、停用蒸馏词。
2. 蒸馏词支持核心、场景、产品三层。
3. 蒸馏词能按来源、层级、优先级筛选。

### Phase G2：周计划和任务改造

优先级：P0

目标：让每个内容任务都能绑定蒸馏词和用户问题。

任务：

1. 扩展 `ContentTask` 字段。
2. 周计划生成时按蒸馏词分配任务。
3. 周计划编辑抽屉支持编辑蒸馏词、用户问题、解决角度。
4. 今日任务页展示蒸馏词字段。
5. 候选池生成任务时带上缺失蒸馏词。

验收标准：

1. 新任务默认带主蒸馏词。
2. 从 GEO 结果生成的任务能说明缺失的蒸馏词。
3. 未绑定蒸馏词的任务有明显提示。

### Phase G3：生成 Prompt 和质检改造

优先级：P0

目标：让生成内容真正围绕蒸馏词写。

任务：

1. 改造 `createDraftWithProviderFallback` 的 Prompt。
2. 本地 fallback 稿件也按蒸馏词结构生成。
3. 扩展 `runDraftQa`，增加 GEO 结构检查。
4. 终稿确认页展示 GEO 质检结果。
5. 保存生成时使用的蒸馏词、Prompt Profile 和证据来源。

验收标准：

1. 生成稿首段能锚定核心蒸馏词。
2. 正文能解释认知逻辑，而不是直接堆产品。
3. 质检能提示缺少蒸馏词、缺少官网引用、缺少品牌绑定。

### Phase G4：GEO 测试和博客诊断改造

优先级：P0

目标：从“提没提到 JOTO”升级为“在哪些认知节点上想到了 JOTO”。

任务：

1. GEO 测试支持选择蒸馏词。
2. 测试结果保存激活概念、缺失概念、竞品占位概念。
3. 博客诊断输出蒸馏词覆盖和引用准备度。
4. 候选池按缺失蒸馏词生成补强任务。

验收标准：

1. GEO 结果能显示回答中激活的蒸馏词。
2. GEO 结果能判断 JOTO 关联强度。
3. 未命中的结果能明确建议补哪类内容。

### Phase G5：周报和内容矩阵复盘

优先级：P1

目标：让周报从任务复盘升级为蒸馏词复盘。

任务：

1. 周报新增蒸馏词覆盖统计。
2. 统计每个核心蒸馏词的内容数量、渠道分布、GEO 命中、官网引用。
3. 输出下周建议：继续写、补强写、减少写、观察。
4. 支持从周报生成下一周蒸馏词任务草稿。

验收标准：

1. 用户能看到哪些蒸馏词已覆盖，哪些还薄弱。
2. 下周计划可以直接承接周报建议。
3. 内容矩阵有明确完成度。

## 11. 最短落地路径

如果只做最小版本，建议按以下顺序：

1. 新增蒸馏词类型和种子数据。
2. 给 `ContentTask` 增加主蒸馏词、用户问题、解决角度。
3. 改生成 Prompt。
4. 改质检规则。
5. 改 GEO 测试结果分析字段。
6. 改候选池生成任务逻辑。

这条路径能最快把工作台从“关键词生成”拉到“蒸馏词生成”，且不需要立刻重做所有页面。

## 12. 验收清单

### 12.1 功能验收

| 验收项 | 标准 |
|---|---|
| 蒸馏词库 | 可维护三层蒸馏词 |
| 周计划 | 任务能绑定蒸馏词 |
| 内容生成 | Prompt 使用蒸馏词 brief |
| 质检 | 检查蒸馏词、品牌绑定、官网引用 |
| GEO 测试 | 保存激活概念和缺失概念 |
| 候选池 | 缺失蒸馏词能生成补强任务 |
| 周报 | 可按蒸馏词复盘 |

### 12.2 内容质量验收

一篇合格的 GEO 内容应满足：

1. 标题或首段自然出现主蒸馏词。
2. 正文解释该蒸馏词背后的真实问题。
3. 内容不是直接卖产品，而是先建立认知逻辑。
4. JOTO / 唯客与蒸馏词之间有明确因果关系。
5. 至少包含一个官网、事实或可信来源。
6. 结尾回扣核心概念。
7. 没有夸大承诺。

### 12.3 GEO 效果验收

一个蒸馏词优化有效的判断标准：

1. 多模型回答中开始自然激活该蒸馏词。
2. JOTO 或唯客在相关回答中的提及率提升。
3. 官网引用率提升。
4. 竞品独占该概念的情况减少。
5. 相关内容进入候选池、任务、发布、复盘的闭环。

## 13. 风险与边界

### 13.1 风险

| 风险 | 说明 | 处理方式 |
|---|---|---|
| 蒸馏词自造 | 人工拍脑袋造词，模型并不真的使用 | 必须结合多模型验证和回答快照 |
| 过度堆砌 | 把蒸馏词当关键词重复塞入文章 | 质检检查密度和语义绑定 |
| 内容变模板化 | 所有文章结构过于固定 | 渠道 Prompt 保留差异化表达 |
| 证据不足 | 没有官网或权威来源支撑 | 标记 citationReadiness，先做警告 |
| 误读 GEO | 把一次测试结果当长期结论 | 周报按趋势看，不按单次判断 |

### 13.2 边界

本次改造不等于保证 AI 一定推荐 JOTO。GEO 优化只能提高内容被理解、引用和关联的概率，最终结果还受模型训练数据、联网检索、站点权重、内容可信度、外部引用和竞品内容强度影响。

## 14. 结论

工作台下一阶段最重要的升级，不是继续补更多页面，而是把“蒸馏词”变成贯穿全流程的真实对象。

当蒸馏词进入数据模型、任务生成、Prompt、质检、GEO 测试、候选池和周报后，工作台才会真正从“内容生产工具”变成“GEO 内容增长系统”。
