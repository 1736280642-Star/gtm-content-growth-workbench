# P0 自动化发布能力与渠道配置说明书

## 1. 文档结论

当前 P0 已完成“正式发布排程与状态治理层”，但尚未完成四个平台的真实正式发布执行。

已经具备：

```text
已确认终稿
-> 创建平台发布排程
-> 到期执行
-> 登录与配置预检查
-> 载荷校验
-> 发布 attempt 记录
-> 发布/验证状态记录
-> 失败原因、下一步和人工接管记录
-> URL 后续 CSV 或人工回填
```

尚未具备：

1. 微信公众号 `freepublish/submit` 与 `freepublish/get` 的真实调用。
2. 掘金、CSDN、知乎的持久浏览器登录、正式发布操作和发布后验证。
3. 常驻发布 Worker 或 Windows Task Scheduler 等系统级触发器。
4. 真实平台环境的端到端验收。

因此，当前 smoke 中出现 `published_pending_url` 只证明 mock 状态机正确，不代表平台已经真实发布文章。

## 2. 当前自动化发布能力

### 2.1 使用的技术与工具

| 能力 | 当前技术/工具 | 当前状态 | 用户影响 |
| --- | --- | --- | --- |
| 工作台与 API | Next.js 14、TypeScript、Route Handlers | 已实现 | 页面和外部任务可以创建、查询、执行发布排程 |
| 状态持久化 | 本地 JSON `data/workbench-state.json` | 已实现，单机试运行 | 可记录 schedule、attempt、失败和 URL；不适合多人并发生产 |
| 平台适配合同 | TypeScript `PublishAdapter` | 已实现 | 四个平台共用 `checkAuth/validatePayload/publish/verify` 合同 |
| 定时排程 | `scheduledAt` + 到期排程 API | 部分实现 | 已能判断到期任务，但仍需要外部 Worker 定时触发 |
| 重复保护 | 同稿件、同平台、同时间排程去重 | 已实现 | 降低重复发布风险 |
| 微信草稿 | Node.js 本机 bridge + 微信官方草稿 API | 已实现草稿，不是正式发布 | 可写入公众号草稿箱，仍需人工正式发布 |
| 掘金/CSDN/知乎草稿 | Node.js bridge + Cookie/内部写作接口 | 实验性草稿能力 | 平台接口易变化，不应作为长期正式发布主路径 |
| 正式发布 mock | 本地 mock adapter | 已实现 | 用于验证状态机，不写入外部平台 |
| 浏览器正式发布 | Playwright 持久浏览器上下文 | 尚未接入 | 掘金、CSDN、知乎暂不能真实自动发布 |
| 发布验证 | 统一验证状态合同 | 合同已实现，真实验证未实现 | 不会把“点击按钮/API 200”直接当成正式发布成功 |
| 人工接管 | `manual_takeover_required` | 已实现状态合同 | 验证码、手机确认、风控出现时停止自动化 |

### 2.2 核心接口

```text
GET  /api/publish-schedules
POST /api/publish-schedules
POST /api/publish-schedules/[id]/run
POST /api/direct-publish
```

创建排程的最小输入：

```json
{
  "publishRecordId": "pub-xxx",
  "platforms": ["wechat", "juejin", "csdn", "zhihu"],
  "scheduledAt": "2026-07-14T10:00:00.000Z",
  "matrixItemId": "monthly-matrix-item-id"
}
```

进入正式发布的终稿必须满足：

1. 稿件状态为 `final`。
2. 终稿质检通过。
3. `distributionAllowed` 不为 `false`。
4. 标题非空。
5. Markdown 正文长度不少于 80 个字符。
6. 发布时间是有效时间。

### 2.3 状态解释

| 状态 | 业务含义 |
| --- | --- |
| `scheduled` | 等待到期执行 |
| `pending_config` | 缺少平台配置或发布能力 |
| `precheck_failed` | 登录态、载荷或平台参数预检查失败 |
| `publishing` | 平台适配器正在执行 |
| `published_verified` | 已确认正式发布，并获得可验证结果 |
| `published_pending_url` | 已确认正式发布，公开 URL 等待后续回填 |
| `pending_verify` | 发布动作已完成，平台仍在审核或等待验证 |
| `manual_takeover_required` | 验证码、手机确认或安全挑战需要人工处理 |
| `failed` | 发布或验证失败 |

## 3. 本地配置规则

### 3.1 配置文件

敏感字段只允许写入项目根目录的本机文件：

```text
.env.local
```

该文件已被 `.gitignore` 忽略。禁止把以下内容写入本文档、Git、聊天记录或截图：

1. AppSecret。
2. Cookie。
3. Access Token、CSRF/XSRF Token。
4. 完整请求 Headers。
5. 浏览器 profile 文件。
6. 私有账号 URL、手机号或账号标识。

### 3.2 模式开关

```dotenv
# 默认不配置时使用 mock，只验证状态机，不写入外部平台
DIRECT_PUBLISH_MOCK=true

# 真实适配器完成并经过 dry-run 验收后才允许开启
DIRECT_PUBLISH_ENABLED=true
```

当前真实 `publish()` 尚未实现，不应仅通过开启 `DIRECT_PUBLISH_ENABLED` 判断真实接入已完成。

## 4. 微信公众号配置

### 4.1 推荐技术路径

```text
官方 API 获取 access_token
-> 上传/读取永久封面素材
-> draft/add 创建草稿
-> freepublish/submit 正式发布
-> freepublish/get 轮询状态
-> 记录文章 ID、URL、时间和失败原因
```

### 4.2 需要获取的字段

| 字段 | 是否必需 | 获取位置/方式 |
| --- | --- | --- |
| `WECHAT_MP_APP_ID` | 必需 | 微信公众平台“设置与开发 -> 基本配置” |
| `WECHAT_MP_APP_SECRET` | 必需 | 同一页面生成或重置，只写本机 `.env.local` |
| `WECHAT_MP_THUMB_MEDIA_ID` | 当前正式预检必需 | 在素材库上传永久图片素材后取得 `media_id` |
| 公网出口 IP 白名单 | 必需条件 | 将运行发布服务机器的公网出口 IP 加入公众号 IP 白名单 |
| `WECHAT_MP_AUTHOR` | 可选 | 默认作者名 |
| `WECHAT_MP_DIGEST` | 可选 | 默认摘要；未配置时由正文生成 |
| `WECHAT_MP_CONTENT_SOURCE_URL` | 可选 | 合法公开来源链接 |
| 评论配置 | 可选 | 是否开启评论、是否仅粉丝评论 |

本地字段：

```dotenv
WECHAT_MP_APP_ID=
WECHAT_MP_APP_SECRET=
WECHAT_MP_THUMB_MEDIA_ID=

WECHAT_MP_AUTHOR=
WECHAT_MP_DIGEST=
WECHAT_MP_CONTENT_SOURCE_URL=
WECHAT_MP_NEED_OPEN_COMMENT=0
WECHAT_MP_ONLY_FANS_CAN_COMMENT=0
```

### 4.3 获取与验收步骤

1. 使用已认证、具备内容发布能力的公众号账号登录管理后台。
2. 在基本配置页面获取 AppID，并生成/重置 AppSecret。
3. 确认调用机器的公网出口 IP，将其加入白名单。
4. 在素材库上传测试封面，取得永久素材 `media_id`。
5. 先验证 access token 获取，不创建文章。
6. 再创建一篇 Kari 已批准的测试草稿。
7. 人工确认草稿内容、封面和排版无误。
8. 调用 `freepublish/submit`，记录返回的发布任务 ID。
9. 轮询 `freepublish/get`，只有平台确认成功才进入 `published_verified` 或 `published_pending_url`。
10. 权限不足、IP 白名单错误或账号不支持发布时记录 `pending_config`，不伪造成功。

## 5. 掘金配置

### 5.1 推荐技术路径

```text
Playwright 持久浏览器 profile
-> 手动完成首次登录
-> 自动检查登录态
-> 校验分类 ID 与标签 ID
-> 打开编辑器并填写内容
-> 点击正式发布
-> 创作中心文章列表/公开页验证
```

现有 Cookie + 内部接口 bridge 只适合草稿实验，不作为长期正式发布主路径。

### 5.2 需要获取的字段

| 字段 | 是否必需 | 获取方式 |
| --- | --- | --- |
| `JUEJIN_CATEGORY_ID` | 必需 | 在编辑器选择分类时，通过页面数据或网络请求确认分类 ID |
| `JUEJIN_TAG_IDS` | 必需 | 在编辑器选择标签时确认标签 ID，英文逗号分隔 |
| 持久浏览器 profile | 必需 | 由专用浏览器上下文首次手动登录后本机保存 |
| `JUEJIN_COOKIE` | 旧 bridge 必需 | 只用于兼容草稿诊断，不建议作为长期正式发布凭证 |
| CSRF、UUID、额外 headers | 按需 | 仅在内部接口草稿诊断确有需要时本机配置 |

当前兼容字段：

```dotenv
JUEJIN_CATEGORY_ID=
JUEJIN_TAG_IDS=

# 以下仅为旧草稿 bridge 兼容字段
JUEJIN_COOKIE=
JUEJIN_CSRF_TOKEN=
JUEJIN_UUID=
JUEJIN_HEADERS_JSON=
```

### 5.3 获取与验收步骤

1. 准备可发布测试文章的掘金账号。
2. 手动进入编辑器，确认目标分类和标签均可选择。
3. 记录分类 ID 和标签 ID 的本地映射，不保存 Cookie 到文档。
4. 使用专用持久 profile 手动登录一次。
5. dry-run 只执行登录检查、编辑器打开、内容填写和参数校验，不点击发布。
6. Kari 确认测试终稿后允许正式点击发布。
7. 在创作中心文章列表确认文章存在，并检查状态是已发布还是审核中。
8. 能打开公开页时记录 URL；审核中则进入 `pending_verify`。
9. 登录失效进入 `auth_required`；验证码或安全挑战进入 `manual_takeover_required`。

## 6. CSDN 配置

### 6.1 推荐技术路径

```text
Playwright 持久浏览器 profile
-> 登录态检查
-> 校验分类、标签、原创类型和公开范围
-> 填写编辑器
-> 正式发布
-> 文章管理列表/公开页验证
```

### 6.2 需要获取的字段

| 字段 | 是否必需 | 获取方式 |
| --- | --- | --- |
| `CSDN_CATEGORIES` | 必需 | 在 CSDN 编辑器确认账号可用分类 |
| `CSDN_TAGS` | 必需 | 确认文章所需标签，使用英文逗号分隔 |
| 持久浏览器 profile | 必需 | 专用浏览器首次手动登录后保存 |
| 文章类型 | 必需业务配置 | 默认建议“原创” |
| 可见范围 | 必需业务配置 | P0 正式发布必须为“公开” |
| `CSDN_COOKIE` | 旧 bridge 必需 | 仅用于现有草稿接口兼容测试 |

当前兼容字段：

```dotenv

CSDN_TAGS=

# 以下仅为旧草稿 bridge 兼容字段
CSDN_COOKIE=
CSDN_HEADERS_JSON=
CSDN_DRAFT_API_URL=
CSDN_DRAFT_PAYLOAD_JSON=
```

### 6.3 获取与验收步骤

1. 使用测试账号登录 CSDN 创作中心。
2. 手动创建一篇测试草稿，确认分类、标签、原创类型和公开范围的真实选项。
3. 建立分类和标签的本地映射。
4. 使用专用持久 profile 保存登录态。
5. dry-run 验证标题、Markdown/HTML 正文、分类和标签可以正确填入。
6. 人工检查无误后执行正式发布。
7. 在文章管理页确认状态不是“草稿”或“仅保存”。
8. 公开页可访问后记录文章 ID 和 URL。
9. 如果平台返回成功但文章仍在审核，进入 `pending_verify`，不能直接标记成功。

## 7. 知乎配置

### 7.1 推荐技术路径

```text
Playwright 持久浏览器 profile
-> 登录态与安全挑战检查
-> 打开文章编辑器

-> 正式发布

```

知乎的验证码、手机确认和安全挑战必须人工接管，不允许绕过平台风控。

### 7.2 需要获取的字段

| 字段 | 是否必需 | 获取方式 |
| --- | --- | --- |
| 持久浏览器 profile | 必需 | 使用专用浏览器上下文首次手动登录后保存 |
| 可写文章/专栏权限 | 必需条件 | 在账号写作后台确认 |
| 可用的人工验证方式 | 必需条件 | 确保账号持有人能处理手机确认或安全挑战 |
| `ZHIHU_COOKIE` | 旧 bridge 必需 | 只用于当前草稿接口兼容诊断 |
| `ZHIHU_XSRF_TOKEN` | 旧 bridge/当前预检需要 | 仅本机保存，会过期 |

当前兼容字段：

```dotenv

ZHIHU_XSRF_TOKEN=
ZHIHU_HEADERS_JSON=

```

### 7.3 获取与验收步骤

1. 准备有文章写作权限的测试账号。
2. 在专用浏览器 profile 中手动登录。
3. 确认账号能够打开写文章页面并创建草稿。
4. dry-run 只填写内容，不点击正式发布。
5. 正式测试前由 Kari 检查终稿内容。
6. 发布后通过创作者中心或公开文章页确认结果。
7. 遇到验证码、手机确认或安全挑战，立即停止并进入 `manual_takeover_required`。
8. 人工完成验证后重新执行预检查，不复用不确定的上一次发布动作，避免重复文章。

## 8. 浏览器 profile 安全规则

掘金、CSDN、知乎的 profile 应放在项目目录之外，例如：

```text
%LOCALAPPDATA%/JotoPublishProfiles/juejin
%LOCALAPPDATA%/JotoPublishProfiles/csdn
%LOCALAPPDATA%/JotoPublishProfiles/zhihu
```

原因：浏览器 profile 内含 Cookie、Local Storage 和账号会话，属于敏感凭证。放在项目外可以降低被 Git、压缩包或同步软件误上传的风险。

后续代码建议增加以下“路径字段”，字段值只写入 `.env.local`：

```dotenv
JUEJIN_BROWSER_PROFILE_DIR=
CSDN_BROWSER_PROFILE_DIR=
ZHIHU_BROWSER_PROFILE_DIR=
```

## 9. 系统级定时任务

当前 `scheduledAt` 只负责记录计划时间，系统仍需一个持续运行的触发器。

推荐本地 P0：

```text
Windows Task Scheduler
-> 每分钟运行 direct-publish worker
-> 查询到期排程
-> 执行带重复保护的发布 attempt
-> 写回状态与失败原因
```

生产环境可以换成常驻队列，但 P0 不需要先引入大型队列系统。

定时 Worker 必须满足：

1. 同一 schedule 不并发执行。
2. 已确认发布不重复执行。
3. 失败保留 retryCount、failureReason 和 nextAction。
4. 人工接管状态不会被自动重试覆盖。
5. 进程退出或机器重启后可以继续扫描未完成排程。

## 10. 推荐真实接入顺序

### 阶段一：微信官方 API

1. 只做配置诊断和 access token 检查。
2. 创建批准的测试草稿。
3. 实现正式发布和状态轮询。
4. 验证 URL、文章 ID 和失败记录。

### 阶段二：三平台浏览器 dry-run

1. 接入 Playwright。
2. 分平台建立持久 profile。
3. 只验证登录、编辑器、分类/标签和内容填写。
4. 不点击发布。

### 阶段三：单平台正式发布

按 `掘金 -> CSDN -> 知乎` 顺序，每次只开放一个平台，并使用单篇批准测试稿。

### 阶段四：定时发布

四平台单次真实发布全部通过后，再接 Windows Task Scheduler 或常驻 Worker。不要在单次发布尚未稳定时直接开放批量定时任务。

## 11. P0 验收清单

每个平台至少完成一次以下闭环：

- [ ] 配置预检查能区分 `ready`、`pending_config`、`auth_required`。
- [ ] dry-run 不写入或发布外部内容。
- [ ] 只发布 Kari 批准的测试稿。
- [ ] 记录平台、schedule ID、attempt ID 和执行时间。
- [ ] 正式发布后通过管理后台或公开页验证。
- [ ] 记录真实平台文章 ID；未知时保持为空。
- [ ] 自动获得 URL 时记录 URL；否则设置 `pendingCsvReturn`。
- [ ] 失败时记录业务可读原因和下一步。
- [ ] 验证码、手机确认、安全挑战进入人工接管。
- [ ] 重试不会生成重复文章。
- [ ] 日志和提交中不存在密钥、Cookie、Token、手机号或私有链接。

## 12. 长期维护判断

1. 微信优先维护官方 API，避免浏览器自动化。
2. 掘金、CSDN、知乎优先维护浏览器 profile 和页面操作合同，Cookie/内部接口只作为诊断兜底。
3. 平台选择器、分类和标签映射应独立配置，不写死在业务页面。
4. 同一类平台变化连续出现三次后，应把修复沉淀成 adapter 测试、诊断脚本和故障 SOP。
5. 月度矩阵只依赖统一发布合同，不直接依赖平台页面实现，避免平台变化导致主业务重构。
