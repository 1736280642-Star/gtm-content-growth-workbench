# V5 内容生产规则链路与动态推广确定性解析方案

版本：2026-07-24

状态：代码实施基线
适用范围：JOTO GTM Workbench V5 月度内容生产

## 1. 结论

V5 内容生产不能继续采用“把所有规则写进一个 Prompt，再让人检查结果”的方式。正式链路应改为：

`知识治理 -> 月度任务冻结 -> EvidencePack -> 生产合同编译 -> 生成前门禁 -> 模型生成与内部自检 -> 确定性机器校验 -> 最多一次自动修复 -> 可用正文 -> 排程与发布 -> 指标回收 -> MonthlyReview`

最终原则：

1. 知识库检索生成的 `FinalEvidencePack` 是正文事实的唯一来源。
2. 通用硬编码只固定执行机制，不写死产品名、身份句、CTA 或 URL。
3. 产品事实、允许表达和禁止表达归产品规则包管理。
4. 内容类型负责回答“这类文章怎么解决问题”，不负责决定推广哪个产品。
5. 渠道规则负责平台结构、链接数量和 CTA 呈现，不保存产品事实。
6. 动态推广配置负责决定“什么条件下推广哪个对象、可使用哪个 CTA 资产”。
7. Prompt 负责指导表达，不承担事实裁决和最终验收。
8. 模型内部自检只提高一次通过率，代码校验才决定正文是否可用。
9. 人工逐段质检取消；产品 Claim、规则激活、关键业务判断和高风险发布仍由人负责。

## 2. 当前代码基线与实施边界

V5 `main` 已有知识治理、Claim、RulePackage、RAG、FinalEvidencePack、PromptGroup、ChannelRule、DraftVersion 和事实追踪等正式能力。当前工作分支尚未完整合入这些模块，因此本轮先落地可以独立测试的领域核心：

1. 动态推广配置契约。
2. 确定性 CTA 解析器。
3. 不可变生产合同编译器。
4. 输出机器校验器。
5. 生成、校验和一次自动修复的编排器。

本轮不伪装已经完成以下能力：

- 真实 MySQL 推广配置仓库和管理 UI。
- WorkBuddy、ADP 或其他产品的正式知识入库。
- OpenSearch 真实检索。
- V5 批量生成页面接线。
- 真实平台公开发布。

领域模块的接口必须保持与正式 V5 对接所需信息一致，后续不得再创建第二套规则语义。

## 3. 规则分层

### 3.1 通用代码硬约束

通用硬约束适用于所有产品、内容类型和渠道：

- 没有可生成的 EvidencePack 时禁止调用模型。
- 每个事实句必须绑定 `evidenceItemId`、`claimId` 和 `sourceRevisionId`。
- 过期、冲突、阻断、无公开权限的证据不能进入公开正文。
- 标题中的数字、能力和结果承诺必须可追溯。
- 禁止泄露 Token、密钥、手机号、账号、客户隐私和私有 URL。
- CTA 只能来自 active 且未过期的推广配置。
- CTA 身份和服务主张必须引用已批准 Claim。
- 模型不能新增 CTA、修改 URL 或改写冻结身份句。
- 同一请求使用幂等键，禁止重复生成和重复发布副作用。
- 输出必须符合固定 Schema。
- 生成后必须运行机器校验。
- 业务规则失败最多自动修复一次；证据缺失和规则冲突不得盲目重试。

硬编码不能包含：

- WorkBuddy、ADP 或任何具体产品名称。
- 固定身份句。
- 固定 CTA 文案。
- 固定服务 URL。
- 固定产品能力或价格。

### 3.2 产品事实规则包

产品规则包保存经过治理的业务事实和表达边界：

- 产品身份与定位。
- 可写能力。
- 条件能力与限制。
- 产品版本、部署、权限和计费边界。
- 禁止主张。
- 竞品和产品关系边界。
- 时效性事实类别。
- 官方引用规则。
- 允许和禁止的内容类型、渠道及证据角色。

产品规则必须关联 Claim 和 SourceRevision。Markdown 文档只能作为录入来源，不能在正式生成时直接充当规则真源。

### 3.3 内容类型配置

内容类型负责写作任务的结构和目标：

- 适用问题与不适用问题。
- 目标读者默认值。
- 内容目标。
- 结构模块。
- 必需章节。
- 篇幅范围。
- 风格特征。
- 案例使用条件。
- 必需证据角色。
- 渠道提示。
- CTA 意图。

内容类型中的 CTA 只能保存意图，例如：

- `none`
- `learn_more`
- `product_evaluation`
- `implementation_assessment`
- `solution_comparison`
- `contact_service`

内容类型不能保存固定身份句或 URL。`coreProblem`、`coreJudgment`、标题和实际读者属于单篇任务快照，不属于内容类型模板。

### 3.4 渠道规则

渠道规则负责平台差异：

- 开头形式。
- 必需结构模块。
- 表格、清单、状态流等内容载体。
- Markdown 格式。
- CTA 呈现模式。
- 官方链接和服务链接数量上限。
- CTA 位置。
- 禁止表达。
- 跨渠道相似度阈值。

渠道规则约束 CTA 的形式，不决定 CTA 的产品和文案。

### 3.5 Prompt 与 Humanizer

Prompt 负责：

- 说明任务目标。
- 注入冻结标题、读者、核心问题和核心判断。
- 注入内容类型和渠道结构。
- 注入 EvidencePack、允许表达、条件表达和禁止表达。
- 注入 Humanizer 指令。
- 要求模型在内部完成起草、自检、改写，只输出最终 JSON。

Humanizer 指令包括：

- 结论前置。
- 删除填充短语和聊天残留。
- 禁止模糊专家背书和虚构引用。
- 避免机械排比、密集粗体和通用乐观结尾。
- 长短句交错。
- 使用具体岗位、动作和业务后果。
- 技术内容解释实现逻辑、底层原因、用户影响和简化方案。

Prompt 不得决定来源优先级、证据是否充分、身份是否合法或 CTA 应该选哪个产品。

### 3.6 生成后机器校验

机器校验至少覆盖：

- JSON 和 Markdown 格式。
- 冻结标题一致性。
- 字数范围。
- 必需章节和内容载体。
- 事实追踪完整性。
- 禁止表达。
- 身份句逐字一致性。
- CTA 数量、位置、URL 和产品相关性。
- URL 公开白名单。
- 敏感信息。
- 重复段落。
- 模型解释或聊天残留。
- 同一选题不同渠道的文本和语义相似度。

主观判断如“小标题是否足够精彩”只进入 Prompt，不作为确定性阻断项。

## 4. 动态推广配置

### 4.1 职责

动态推广配置回答：

> 这篇文章在什么条件下，可以推广哪个产品或服务，使用哪条批准后的 CTA，链接到哪个公开入口。

它不保存产品事实，不决定文章结构，也不替代渠道规则。

### 4.2 PromotionProfileVersion

建议字段：

```ts
interface PromotionProfileVersion {
  promotionProfileVersionId: string;
  version: number;
  status: "draft" | "active" | "superseded";
  targetEntityIds: string[];
  excludedEntityIds: string[];
  applicableProductGroups: string[];
  articleScope: "single_product" | "multi_product" | "comparison" | "brand";
  promotionGoal: string;
  ctaIntent: CTAIntent;
  applicableContentTypes: string[];
  applicableTitleCategories: string[];
  allowMultiProduct: boolean;
  requiresPrimaryEntity: boolean;
  priority: number;
  validFrom?: string;
  validUntil?: string;
  variants: PromotionCtaVariant[];
  approvedBy?: string;
  approvedAt?: string;
}
```

CTA 变体字段：

```ts
interface PromotionCtaVariant {
  ctaVariantId: string;
  channel: string;
  label: string;
  publicUrl: string;
  identityClaimIds: string[];
  serviceClaimIds: string[];
  allowedRenderModes: string[];
  status: "active" | "disabled";
}
```

`CTA 是否必需`不放在推广配置中。它由内容类型规则生成并冻结为任务的 `promotionRequired`，避免内容类型与推广资产各保存一套互相冲突的必需性开关。

### 4.3 确定性解析

CTA 选择必须在正文生成前完成：

`问题 -> 产品实体识别 -> 冻结 targetEntityIds/primaryEntityId -> 内容类型 CTA intent -> 渠道 -> active PromotionProfile -> Claim 与有效期校验 -> 明确优先级 -> CTAPlan`

排序采用字典序规则，不采用模糊加权总分：

1. 精确产品匹配优先于产品组匹配。
2. 精确渠道优先于通配渠道。
3. 精确 CTA intent 优先于通用意图。
4. 精确内容类型优先于不限类型。
5. 主产品优先于次要产品。
6. 再按人工配置的 `priority` 排序。
7. 同等优先级仍有多个结果时阻断，禁止随机选择。

找不到 CTA 时：

- 普通内容且 CTA 非必需：生成无 CTA 正文。
- 推广任务且 CTA 必需：生成前阻断。
- 禁止回退到固定 WorkBuddy CTA、品牌默认 CTA 或模型临时编写。

### 4.4 CTAPlan

解析输出必须冻结：

```ts
interface CTAPlan {
  promotionProfileVersionIds: string[];
  targetEntityIds: string[];
  selectedVariants: ResolvedCtaVariant[];
  renderMode: string;
  maxCtaCount: number;
  selectionReasons: string[];
  planHash: string;
}
```

模型不能修改 CTA 对象、身份句、服务范围、URL 和数量。

## 5. ProductionContractSnapshot

正式生成前编译不可变生产合同：

```text
TaskSnapshot
+ FinalEvidencePack
+ ProductRulePackageVersion
+ ArticleTypeProfileVersion
+ ChannelRuleVersion
+ ExpressionProfileVersion
+ PromotionProfileVersion
+ CTAPlan
+ ValidatorPolicy
= ProductionContractSnapshot + contractHash
```

规则合并算法：

- 允许能力取交集。
- 禁止表达取并集。
- 必需章节取并集。
- 字数范围取内容类型和渠道规则的交集。
- CTA 先按产品和意图筛选，再应用渠道数量上限。
- 官方链接只能来自 EvidencePack 或已批准推广配置。
- 合并结果无有效字数范围、存在规则冲突或 CTA 缺少 Claim 时，生成前阻断。

`contractHash` 必须覆盖所有输入版本和最终结构化合同。生成运行只引用冻结合同，不读取实时变化中的配置。

## 6. 生成与恢复状态机

```text
preflight
  -> awaiting_material
  -> rule_conflict
  -> ready_for_generation

ready_for_generation
  -> generating
  -> validating
  -> repairing_once
  -> available
  -> failed
```

错误处理：

| 错误 | 行为 |
| --- | --- |
| `evidence_missing` | 不调用模型，创建知识待办 |
| `rule_conflict` | 不调用模型，进入规则维护 |
| `promotion_required_missing` | 不调用模型，补推广配置 |
| `provider_failure` | 技术重试最多三次 |
| `fixable_output` | 携带明确错误自动修复一次 |
| `still_invalid` | 失败并保留上一版可用正文 |
| `sensitive_output` | 立即阻断，不自动发布 |

自动修复只能修复格式、篇幅、章节、CTA 和禁用表达，不能新增证据或改变产品对象。

## 7. 多渠道相似度

同一问题的多渠道正文不能只换标题。相似度校验分两层：

1. 确定性文本检查：标准化后长片段、段落指纹和重复段落比例。
2. 可选语义检查：使用固定版本 Embedding 对正文计算相似度。

阈值属于版本化渠道策略，不写死在 Prompt。超阈值时允许一次基于目标渠道结构的重写；仍超阈值则失败。

产品事实相同不应被判重复，校验应忽略 CTA、官方引用和短事实句，主要比较叙事段落和结构序列。

## 8. 人机边界

取消的人工步骤：

- 逐段检查禁用词。
- 手工核对字数、链接数量和固定结构。
- 手工把同一类机器错误逐条改回。

保留的人工判断：

- 产品实体歧义确认。
- Claim、来源冲突和高风险事实审批。
- 产品规则包、渠道规则和推广配置激活。
- 月度核心问题、核心判断和推广目标批准。
- 真实公开发布确认，直到外部平台适配器完成稳定验收。

## 9. 代码模块

本轮新增：

```text
src/lib/v5/
  content-production-contracts.ts
  promotion-resolver.ts
  production-contract-compiler.ts
  production-output-validator.ts
  content-production-service.ts
```

模块职责：

- `content-production-contracts.ts`：领域契约和错误码。
- `promotion-resolver.ts`：CTA 候选过滤、排序、冲突阻断和 CTAPlan 冻结。
- `production-contract-compiler.ts`：合并任务、证据、内容类型、渠道和推广规则。
- `production-output-validator.ts`：确定性输出校验。
- `content-production-service.ts`：生成、校验、一次修复和失败分类编排。

## 10. 测试策略

最低合约测试：

1. 无 EvidencePack 时模型调用次数为零。
2. CTA 非必需且无匹配配置时生成无 CTA 合同。
3. CTA 必需且无匹配配置时生成前阻断。
4. 同产品多 CTA 按渠道、意图和优先级确定性选择。
5. 同优先级冲突时阻断。
6. 多产品任务受渠道最大 CTA 数限制。
7. CTA 缺少 Claim 时阻断。
8. 字数范围无交集时阻断。
9. 模型新增 URL、修改 CTA 或出现禁用词时校验失败。
10. 事实追踪不完整时失败。
11. 首次可修复失败后只修复一次。
12. 第二次仍失败时不返回可用正文。
13. 敏感信息命中时阻断。
14. 相同输入生成相同 `CTAPlan.planHash` 和 `contractHash`。

## 11. 分阶段执行计划

### P0：领域合同与确定性解析

- 建立通用契约。
- 实现 PromotionProfile 和 CTAPlan。
- 实现确定性解析与冲突错误。
- 建立合约测试。

验收：新增产品和 CTA 只增加配置，不修改解析代码。

### P0：生产合同与生成前门禁

- 编译内容类型、渠道和 EvidencePack。
- 生成不可变合同和 Hash。
- 在 Provider 调用前返回明确阻断原因。

验收：证据、规则或推广配置不完整时模型调用次数为零。

### P0：输出校验与一次修复

- 实现字数、结构、事实、CTA、URL、敏感信息和重复检查。
- 将机器错误转为稳定错误码和修复指令。
- 最多一次自动修复。

验收：不合格正文不会进入 available。

### P1：正式 V5 持久化和 UI

- 增加 MySQL PromotionProfileVersion、ProductionContractSnapshot。
- 接入 FinalEvidencePack、GenerationRun 和 DraftVersion。
- 增加推广配置管理界面和月度任务 CTA 预览。
- 批量生成页展示阻断原因和确定性 CTA 选择依据。

### P1：发布和复盘

- 接入平台草稿箱和真实发布状态。
- 校验公开 URL。
- 回收渠道指标。
- 同一规则累计三次失败自动创建维护任务。
- MonthlyReview 提出规则调整建议，由人批准后生效。

## 12. 完成定义

只有同时满足以下条件，才能宣称真实完整链路完成：

- 正式知识来源可导入、治理、索引和检索。
- FinalEvidencePack 是唯一事实输入。
- 产品、内容类型、渠道和推广配置均为版本化真源。
- 生产合同可追溯、不可变、可复现。
- 生成前门禁、输出校验和一次修复真实执行。
- 不合格正文不会交给用户手工补救或进入发布。
- DraftVersion 能进入排程和真实发布记录。
- 平台返回可验证公开 URL。
- 指标进入 MonthlyReview 并形成下一月建议。
