## 2. 项目定位

JOTO GTM 内容工作台不是单纯的 AI 写作工具，也不是继续做深一个通用 GEO 系统。

它的 MVP 目标是服务 JOTO 当前真实的市场内容工作：

1. 自动化生成渠道文章选题与内容。
2. 支持人工确认终稿后进入发布队列。
3. 对官网博客进行 SEO、GEO、AI Bot、AI 引用测试等监控与诊断。
4. 将渠道数据与官网博客诊断结果反哺下一周选题。
5. 为后续接入官网博客创作模块预留入口。

MVP 阶段聚焦两个对象：

1. JOTO 官方品牌传播。
2. 唯客 AI 护栏单一产品内容自动化。

当前阶段不把官网博客创作作为主流程。官网博客只做监控、分析判断和建议；后续可从博客候选池进入“博客创作任务”模块。

## 3. 已形成的核心判断

### 3.1 为什么不继续做深 GEOFlow

经过 GEO 尝试后，当前判断是：继续做深一个通用 GEO 系统的投入产出比不如做一个更宽的 GTM 内容工作台。

原因包括：

1. GEOFlow 作为通用 GEO 产品偏重，超过当前内部市场工作的真实需要。
2. 豆包、DeepSeek、ChatGPT 等 AI 平台账号和测试成本有限。
3. 清除用户记忆、重复提问、人工记录的代价较高。
4. 独立信源站当前更像测试资产，不应成为 MVP 工作台模块。
5. AI 当前更可能引用官网博客，而不是渠道文章或独立信源站。
6. 渠道文章主要给人看，负责触达、验证选题和传播；官网博客主要给 AI 和搜索看，负责信源、SEO/GEO 和权威内容沉淀。

### 3.2 渠道文章与官网博客的分工

| 内容类型 | 主要对象         | 核心职责                     | 主要指标                                   |
| ---- | ------------ | ------------------------ | -------------------------------------- |
| 渠道文章 | 人            | 触达用户、验证选题、建立认知、获得互动      | 阅读、浏览、点赞、收藏、评论、转发、咨询、官网点击              |
| 官网博客 | AI + 搜索 + 用户 | 成为稳定信源、承接 SEO/GEO、沉淀权威内容 | 抓取、收录、AI Bot PV、AI 提及、AI 引用、关键词覆盖、内容结构 |

AI Bot PV 只能说明 AI 爬虫访问过页面，不能说明 AI 已经引用页面。GEO 数据链路应区分：

```text
被发布
-> 被抓取
-> 被收录或进入索引
-> 被 AI 检索到
-> AI 回答中提及 JOTO / 唯客
-> AI 回答引用官网博客链接
```

## 4. MVP 主流程

当前确认的 MVP 主流程：

```text
周计划配置
-> 自动生成渠道选题与任务
-> 批量创作当天渠道文章
-> 人工确认终稿
-> 进入发布队列
-> 发布后 URL 回填
-> 渠道数据与官网博客 SEO/GEO 诊断回流
-> 反哺下周选题
```

官网博客流程：

```text
抓取官网博客
-> SEO 基础检查
-> AI Bot 访问分析
-> GEO Prompt 测试
-> 内容缺口判断
-> 博客优化建议
-> 博客候选池
-> 博客创作任务入口（待接入）
```

博客创作不进入当前 MVP 主流程，只保留入口。
